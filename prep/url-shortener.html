Design a URL shortener

Like TinyURL or bit.ly: turn https://a-very-long-link.com/… into short.ly/xY7aB2, and redirect anyone who visits the short link back to the original. We'll build it from first principles until it actually clicks.

Start here

Why this question is everywhere

A URL shortener looks trivial — "just store a mapping" — and that's exactly why interviewers love it. The simple core lets them keep asking "okay, now make it handle a billion redirects a day" and watch how you reason about scale, IDs, caching, and data partitioning.

Almost every sub-skill of system design shows up here:

If you remember one thing

A URL shortener is fundamentally a distributed hash map: short code → long URL. Writes are rare, reads are enormous, and reads must be fast. Every design decision flows from that one sentence. Say it out loud at the start of the interview.

The core insight

It's a key-value lookup. That's it.

Strip everything away and the whole product is two operations on a dictionary:

# write (rare)
db["xY7aB2"] = "https://some-very-long-url.com/path?x=1"

# read (constant, massive)
long_url = db["xY7aB2"]   # → then HTTP redirect the browser there

Everything hard about the problem is a consequence of three pressures on that tiny dictionary:

  1. The key must be short and unique. How do you mint a brand-new 7-character code, billions of times, with no collisions, across many servers? → the ID generation problem.
  2. The reads are gigantic and latency-sensitive. A click should redirect in milliseconds. → caching + read replicas.
  3. The data outgrows one machine. Billions of rows. → sharding.

Hold that frame. Now we build it the way you'd build it live in the interview.

Step 1

Clarify requirements (do this first, always)

Never start drawing boxes. Spend the first 2–3 minutes scoping. It signals seniority and stops you from designing the wrong thing.

Functional requirements (what it does)

  • Shorten: given a long URL, return a short URL.
  • Redirect: given a short URL, send the user to the original.
  • (Probe these): custom aliases (short.ly/my-brand)? Link expiry? Analytics (click counts)? User accounts?

Non-functional requirements (the qualities that shape the design)

  • High availability. If redirects are down, every existing link is broken. This is the scariest failure.
  • Low latency. Redirects must feel instant (< ~100ms).
  • Massive read scale, read:write ≈ 100:1 (people click far more than they create).
  • Short codes — as short as reasonable.
  • Not easily guessable (sometimes — depends on whether links are "secret").
Questions to actually ask the interviewer

"How many URLs created per day?" · "What's the read-to-write ratio?" · "How long should links live — forever?" · "Do we need custom aliases or analytics?" · "Should codes be unpredictable?" Their answers define your numbers and your data model. Pretend they said: 100M new URLs/day, 100:1 reads, links live ~5 years, analytics yes-but-later.

What these numbers prove

Step 3

The API

Two endpoints. Keep it boring.

# create a short URL
POST /api/shorten
   body:  { "longUrl": "https://example.com/very/long/path",
            "customAlias": "my-link"   # optional,
            "expiresAt": "2030-01-01"     # optional }
   → 201 { "shortUrl": "https://short.ly/xY7aB2" }

# the redirect — this is the hot path, 100× the traffic
GET /{shortCode}
   → 301/302 redirect  →  Location: https://example.com/very/long/path
Note the asymmetry

The read endpoint is just GET /{shortCode} at the root — short and clean, because it's what users paste and click billions of times. The write endpoint is a normal JSON API. Already the design reflects "reads dominate."

Step 4

The data model

One core table/collection. The access pattern is "look up by short code," so the short code is the primary key.

SQL or NoSQL here?

Why NoSQL is the common answer

The access pattern is a pure key lookup by short_code — no joins, no complex queries. We need it to scale horizontally to 91 TB / 100B+ rows. A key-value / wide-column store (DynamoDB, Cassandra) is built exactly for this: huge scale, simple lookups, easy sharding by key. SQL works (and is fine to start), but it scales out less naturally. Say: "I'd reach for a key-value store because the access pattern is a single-key lookup at extreme scale — but SQL with sharding is a reasonable v1."

The heart of it

How short is the code? Base62 math

This is the part interviewers actually want to see. How do we make codes that are short, unique, and use URL-safe characters?

Base62: the alphabet

Use the 62 URL-safe characters: a–z (26) + A–Z (26) + 0–9 (10) = 62. Each character of the code is one "digit" in base 62. So a code of length n can represent 62ⁿ distinct values:

625  =        916,132,832   (~0.9 billion)
626  =     56,800,235,584   (~56 billion)
627  =  3,521,614,606,208   (~3.5 trillion)
The conclusion

We estimated ~182 billion URLs in 5 years. 62⁶ ≈ 56B isn't quite enough; 62⁷ ≈ 3.5 trillion is plenty with huge headroom. So a 7-character code is the answer. (bit.ly and friends use ~7.) Being able to derive "7 characters" from your traffic estimate is a chef's-kiss moment in the interview.

Why base62 and not, say, hex or base64?

  • Hex (base16) would need far more characters for the same range (16⁷ is tiny) → longer codes.
  • Base64 adds + and / (and = padding) — not URL-safe without escaping.
  • Base62 is the sweet spot: only letters+digits, all URL-safe, short codes.
The heart of it

Generating the code: 3 approaches

We know codes are 7 base62 chars. But how do we pick the next one so it's unique — especially across many servers? This is the make-or-break discussion.

Approach A · Hash the long URL

Run the long URL through a hash (MD5/SHA-256), take the first ~7 base62 chars.

Pros

  • Stateless — no counter to coordinate.
  • Same URL → same code (free dedup).

Cons

  • Collisions: two different URLs can hash to the same 7 chars. Must check the DB and re-hash (append a salt) on conflict.
  • Truncating a hash wastes its uniqueness; collision rate grows as you fill up.

Approach B · Counter + base62 encode most recommended

Keep a single global counter. Every new URL gets the next integer; encode that integer to base62 and you have the code.

id = 1           → base62 → "1"
id = 125         → base62 → "21"
id = 100000000000 → base62 → "1L9zO9O"   (7 chars)
Why this is elegant

The counter is monotonic and unique by construction — collisions are literally impossible. Base62-encoding an ever-increasing integer just walks through the code space in order. No DB lookup to check for conflicts on write. The only hard part: making the counter work across many servers (next).

The catch: codes become guessable

Sequential IDs mean code xY7aB3 is "next to" xY7aB2 — someone can enumerate your links. Fixes: encode with a secret permutation/bijection of the ID space, or interleave/scramble bits. Mention this tradeoff; it shows depth.

Approach C · Key Generation Service (KGS)

A standalone service pre-generates millions of unique random codes offline and stores them in a "available keys" table. On a write, an app server just grabs one and marks it used. Generation is decoupled from the request path entirely.

  • Pro: no collision checks on the hot path, no sequential-guessing problem, codes can be random.
  • Con: the KGS is a component to build + make highly available; needs careful concurrency so two servers never hand out the same key (give each app server a block of keys to avoid contention).

The real challenge: a unique counter across many servers

Approach B needs one source of truth for "the next number," but a single counter is a bottleneck and a single point of failure. Options:

TechniqueHowTradeoff
Ranges / blocks cleanA coordinator hands each app server a block of IDs (e.g. 1–1000, then 1001–2000). Servers mint locally from their block, fetch a new block when empty.Almost no coordination on the hot path; gaps if a server dies (fine). The standard answer.
Redis INCRAtomic increment on a single fast in-memory counter.Simple; Redis becomes a critical dependency (replicate it).
Snowflake-style IDCompose a unique 64-bit ID from timestamp + machineId + sequence — no coordination at all.IDs are bigger; codes a touch longer. Used by Twitter/Instagram.
DB auto-incrementLet one SQL DB own the sequence.Simplest, but a write bottleneck + SPOF at scale.
What to say

"I'd use a counter encoded to base62 for guaranteed-unique codes, and distribute the counter by handing each app server a block of IDs from a coordinator — so there's no per-request contention. If guessability matters, I'd scramble the ID before encoding. Hashing and a KGS are valid alternatives with different tradeoffs." That single paragraph covers the whole topic.

Cache-aside, the pattern to name

On a miss we read the DB and populate the cache so the next click is a hit. Because clicks follow the 80/20 rule (a few links go viral, most are dormant), a relatively small cache absorbs the overwhelming majority of the 116k reads/sec. The cache is what makes the read path survive.

The senior answer

"I'd use 302 if click analytics matter — every visit hits our server so we can count it. If we care more about offloading traffic and don't need exact counts, 301 lets browsers cache the redirect and cuts our load. It's an analytics-vs-load tradeoff."

Scaling

Caching — the read-scale lever

The redirect path is 100× the writes, so caching isn't optional — it's the primary defense.

  • What to cache: short_code → long_url. Small, simple, immutable once created (mappings rarely change), so caching is safe and effective.
  • Pattern: cache-aside (lazy) — on miss, read DB, populate cache.
  • Eviction: LRU — keep the hottest links in memory, drop the cold ones. Fits the 80/20 click distribution perfectly.
  • TTL: optional; mappings are stable so you can cache aggressively. Set TTL to bound memory and respect expiry.
  • Where: a Redis/Memcached tier; plus a CDN/edge cache can serve very hot redirects close to users.
Tie-back to Argus

This is the exact cache-aside + LRU pattern from your fundamentals page — and conceptually the same as RTK Query caching responses on your frontend. Mappings being immutable is what makes the cache so easy here: no invalidation problem, the hardest part of caching, mostly disappears.

Scaling

Scaling the database: replication & sharding

Replication (for read throughput + availability)

Run a primary for writes and multiple read replicas. Redirects (reads) hit replicas; shortens (writes) hit the primary. Replicas also give failover. Watch for replication lag — but mappings are immutable, so a slightly-stale replica is basically harmless here (a brand-new link might 404 for a few ms before it propagates — acceptable, or route fresh reads to primary briefly).

Sharding (for storage — we have 91 TB)

No single machine holds 91 TB / 100B+ rows, so partition the data across shards. The question is the shard key:

StrategyHowVerdict
Hash of short_code goodshard = hash(shortCode) % NEven distribution; a lookup goes straight to the right shard (we always know the code). The natural fit.
Range-basedcodes A–M on shard 1, N–Z on shard 2…Risks hotspots if creation isn't uniform.
Why hash-by-short_code just works

Every read is "look up this exact code." Hashing the code to pick the shard means the redirect path computes the shard in O(1) and hits exactly one machine — no scatter-gather. The lookup key and the shard key are the same thing, which is the ideal sharding scenario.

The resharding trap — mention consistent hashing

Plain % N means adding a shard remaps almost everything. Consistent hashing moves only a small fraction of keys when N changes — the standard fix. Naming this when discussing sharding is a strong signal.

Follow-ups

The deep-dive questions they'll tack on

Once the core is solid, interviewers pick one of these to go deeper. Have a one-paragraph answer for each.

Custom aliases

User wants short.ly/my-brand. On write, check if the alias exists (it must be unique); if free, store it like any code; if taken, return 409. Custom aliases live in the same table — they're just codes you didn't auto-generate.

Collisions (if using hashing)

Two URLs hash to the same code. On insert, detect the conflict (unique constraint), then re-hash with a salt or append random chars and retry. The counter approach sidesteps this entirely — worth restating as the reason you preferred it.

Link expiry & cleanup

Store expires_at; the read path checks it and returns 404 if past. Reclaim space with a background job (lazy deletion) or a TTL index in the DB — don't delete synchronously on the hot path.

Analytics (click counts)

Don't increment a counter in the DB on every redirect — that's a write on the hottest path and will melt. Instead fire an event (to Kafka / a queue) asynchronously and aggregate offline. The redirect returns immediately; analytics are eventually consistent. This is a great place to show you protect the hot path.

Rate limiting & abuse

Limit shortens per user/IP (token bucket) to stop spam. Screen long URLs against a malware/phishing blocklist before storing — a shortener is a juicy tool for hiding malicious links.

High availability

No single points of failure: multiple app servers behind the LB, replicated cache, replicated + sharded DB, replicated ID service. The redirect path degrades gracefully — even if writes are down, existing links must keep resolving.

Recap

How to present this in 5 minutes

The order to walk through it live, so you never blank:

  1. Frame it: "It's a distributed key-value store, code → URL, read-heavy, reads must be fast."
  2. Clarify: ask scale, read:write, expiry, custom aliases, analytics.
  3. Estimate: derive QPS, storage, and 7-char base62 code from the numbers.
  4. API + model: POST /shorten, GET /{code}; one KV table keyed by code.
  5. Core problem: code generation → counter + base62, distributed via ID blocks; mention hashing/KGS alternatives + the guessability tradeoff.
  6. Read path: cache-aside + Redis (LRU), 301-vs-302 tradeoff.
  7. Scale: stateless app servers + LB, DB replicas + shard by hash(code), consistent hashing.
  8. Deep dives: async analytics, expiry job, rate limiting, HA.

Flashcards

Rapid-fire: URL shortener

How long should the short code be, and why?

7 base62 characters. 62⁷ ≈ 3.5 trillion combinations, comfortably above our ~182B-in-5-years estimate. 62⁶ (~56B) would run out. Base62 = a–z, A–Z, 0–9, all URL-safe.

How do you generate unique codes across many servers?

Use a global counter encoded to base62 (unique by construction — no collisions). Distribute it by having a coordinator hand each app server a block of IDs, so servers mint locally with no per-request contention. Alternatives: hashing (needs collision handling), a Key Generation Service (pre-generated keys), or Snowflake-style IDs.

301 or 302 redirect?

302 (temporary) if you want click analytics — every visit hits your server. 301 (permanent) if you want to offload load — browsers cache the redirect and may skip your server, but you lose analytics. It's an analytics-vs-load tradeoff.

How do you handle 116k reads/sec?

Cache-aside with Redis (LRU eviction) absorbs most reads — clicks follow an 80/20 distribution, so a modest cache yields high hit rates. Behind it, read replicas spread DB load and a CDN can serve very hot redirects at the edge. Mappings are immutable, so caching is safe with no invalidation headache.

How do you shard the database?

Shard by hash(short_code). Since every lookup is by exact code, hashing the code picks the shard in O(1) and hits exactly one machine — the lookup key is the shard key. Use consistent hashing so adding shards remaps only a small fraction of keys.

How do you count clicks without melting the hot path?

Don't write to the DB on each redirect. Emit an async event to a queue/stream (e.g. Kafka) and aggregate offline. The redirect returns immediately; analytics are eventually consistent. Protecting the hot path from synchronous writes is the key idea.

SQL or NoSQL?

NoSQL key-value/wide-column (DynamoDB, Cassandra) fits best: the access pattern is a single-key lookup at extreme scale with no joins, and it shards naturally. SQL with sharding is a fine v1, but scales out less gracefully.
