The one mental model

LangGraph is a state machine for LLM workflows. You declare nodes (functions), edges (what runs next), and one shared state object. The runtime drives the loop: it picks the next node, calls it, merges what it returns into the state, checkpoints, and repeats until it reaches END.

Say this and you've framed it

"It's not a chain — it's a directed graph with a typed shared state and one conditional edge that can loop. Each node is a pure-ish async function that reads the state and returns a partial update; the framework merges updates (last-write-wins, or via a reducer for channels that accumulate) and snapshots the whole state after every super-step, which is what makes it resumable."

Three primitives carry the whole design — internalize these names:

PrimitiveIn ArgusWhy it matters
State (channels)GraphState — a TypedDictThe single source of truth threaded through every node
Reducer_merge_unique_sources on raw_sourcesLets a channel accumulate across the loop instead of clobbering
Conditional edgeroute_after_qualityThe self-correction: loop back for more research, or finalize

The rest of this page is those three ideas, plus the production concerns layered on top (DI, failure, checkpointing).

The graph

The one-breath summary

"Plan the questions → research the web → extract structured signals → analyse into prose → gate on quality → either loop back to plan sharper questions, or report." Six nodes, one branch, one accumulator.

The graph · code

How the graph is assembled

The whole topology is built declaratively in one function. Note the three moves: add nodes (with a retry policy), wire the static edges, then add the one conditional edge.

# backend/app/workflow/graph.py
def build_graph(deps: WorkflowDeps, *, checkpointer=None):
    retry = RetryPolicy(
        max_attempts=deps.settings.workflow_node_retry_limit,
        retry_on=_retry_on,            # transient transport errors only
    )
    builder = StateGraph(GraphState)   # the state schema IS the graph's type

    nodes = {
        "planner": build_planner(deps),
        "researcher": build_researcher(deps),
        "signal_extractor": build_signal_extractor(deps),
        "analyst": build_analyst(deps),
        "quality_check": build_quality_check(deps),
        "reporter": build_reporter(deps),
    }
    for name, fn in nodes.items():
        builder.add_node(name, fn, retry_policy=retry)

    # static backbone
    builder.add_edge(START, "planner")
    builder.add_edge("planner", "researcher")
    builder.add_edge("researcher", "signal_extractor")
    builder.add_edge("signal_extractor", "analyst")
    builder.add_edge("analyst", "quality_check")

    # the one branch: loop back to planner, or go to reporter
    max_iterations = deps.settings.workflow_max_research_iterations
    builder.add_conditional_edges(
        "quality_check",
        lambda state: route_after_quality(state, max_iterations=max_iterations),
        {"planner": "planner", "reporter": "reporter"},
    )
    builder.add_edge("reporter", END)

    return builder.compile(checkpointer=checkpointer)
Three things to point out unprompted

(1) The state schema (GraphState) is passed to StateGraph — the graph is typed by it. (2) Every node gets the same RetryPolicy, but it only retries transient errors. (3) The checkpointer is injected at compile() time — pass a real one in prod, omit it in tests. Same graph, swappable durability.

State

The shared, typed state

Every node reads and writes one GraphState. It's a TypedDict(total=False) — all keys optional, filled in as the run progresses. Three families of fields: inputs, accumulating outputs, and control.

# backend/app/workflow/state.py
class GraphState(TypedDict, total=False):
    # --- inputs (seeded by the runner) ---
    session_id: str
    run_id: str
    company_name: str
    website: str
    objective: str

    # --- outputs, one per node ---
    plan: list[SubQuestion]
    raw_sources: Annotated[list[RawSource], _merge_unique_sources]  # appended, deduped
    extracted_signals: list[BusinessSignal]
    analysis: AnalysisBlock
    quality: QualityVerdict
    report: dict[str, Any]                  # final ReportDraft dump

    # --- control ---
    retry_counts: dict[str, int]
    errors: Annotated[list[dict[str, Any]], add]          # append-only log
    research_iteration: int                               # bumped per loop
    researched_question_ids: Annotated[list[str], add]    # what's been chased
The serialization constraint — say it before they ask

Everything in the state must be JSON / msgpack-serializable, because the whole dict is snapshotted into Mongo after every super-step. So no live handles — no DB clients, no sockets, no queues — may live in the state. Those travel separately (via a RunContext bound to a contextvars.ContextVar). This is the single most important constraint in the design.

The TypedDict gives you editor/type-checker safety without the overhead of a class per node. Each node returns a plain dict whose keys are a subset of these — never the whole state.

State · the killer detail

Reducers: how a channel accumulates

This is the detail that proves you understand LangGraph rather than just used it. A node returns a partial dict; LangGraph merges it into the running state. How it merges depends on the channel.

Channel kindMerge behaviourExample
PlainLast-write-wins — the new value replaces the oldplan, analysis, quality
Annotated[T, reducer]Reducer(old, new) — a merge function combines themraw_sources, errors, researched_question_ids

The custom reducer on raw_sources appends new sources but de-dupes by URL, so a second research loop adds to the first batch instead of overwriting it:

def _merge_unique_sources(left: list[RawSource], right: list[RawSource]):
    """Reducer for raw_sources: append new, de-dup by URL."""
    seen = {src["url"] for src in left}
    merged = list(left)
    for src in right:
        if src["url"] in seen:
            continue
        seen.add(src["url"])
        merged.append(src)
    return merged
Why this single annotation carries the loop

When quality_check loops back, the researcher runs again and returns more sources. A plain channel would clobber the first batch — the loop would forget everything it found. The reducer keeps the union, deduped by URL. That one Annotated[...] is what makes the research loop accumulate instead of forget. (errors and researched_question_ids use the simpler operator.add reducer — plain list concatenation.)

For the full node-by-node picture of the state growing, see the "Watch the state grow" walkthrough on the main prep page — this page focuses on the machinery; that one focuses on the data.

The loop · crown jewel

The conditional edge — and why it loops to the planner

This is the part that makes it an agent, not a pipeline: it inspects its own output quality and decides whether to do more work. The routing function is deliberately pulled out of the node so it's unit-testable in isolation.

# backend/app/workflow/graph.py
def route_after_quality(state, *, max_iterations) -> Literal["planner", "reporter"]:
    quality = state.get("quality")
    iteration = state.get("research_iteration", 0)
    if (
        quality
        and quality.get("needs_more_research")
        and quality.get("missing_areas")        # must have a concrete target
        and iteration < max_iterations          # bounded — no infinite loop
    ):
        return "planner"
    return "reporter"
Why loop to the planner, not the researcher

Looping straight to the researcher would re-run it against the already-researched plan — every question id is already in researched_question_ids, so it would gather nothing new and burn tokens spinning. Routing to the planner lets it read quality.missing_areas and emit fresh gap-closing sub-questions, giving the next research pass something genuinely new to chase. (This is your commit a094b5a — a real bug you found and fixed.)

The three guards, and why each exists

(1) needs_more_research — the gate's own verdict. (2) missing_areas non-empty — the schema permits needs_more_research=true with an empty gap list; without a target the planner would just regenerate its original questions and loop with zero progress. (3) iteration < max_iterations — a hard cap (workflow_max_research_iterations) so a stubborn gate can't loop forever. There's also a LangGraph-level recursion_limit as a final backstop.

How the planner uses the gap on a follow-up pass

The planner detects it's a follow-up (existing plan + missing areas) and appends sharper questions rather than starting over:

# backend/app/workflow/nodes/planner.py
existing = state.get("plan") or []
missing  = quality["missing_areas"] if quality else []
is_followup = bool(existing) and bool(missing)

user = planner_user_with_gaps(..., missing) if is_followup else planner_user(...)
result = await deps.llm.complete(PLANNER_SYSTEM, user, response_model=PlanResult, model=model)

start_idx = len(existing)                # new ids continue the numbering: q2, q3…
new_questions = [SubQuestion(id=f"q{start_idx + i}", ...) for i, q in enumerate(result.questions)]
return {"plan": [*existing, *new_questions]}

And the researcher only chases the new ids, so the loop never re-fetches what it already has:

# backend/app/workflow/nodes/researcher.py
already = set(state.get("researched_question_ids") or [])
todo = [q for q in plan if q["id"] not in already]   # only the fresh questions
...
return {
    "raw_sources": deduped,                          # merged via the reducer
    "research_iteration": iteration + 1,             # bump the loop counter
    "researched_question_ids": [q["id"] for q in todo],  # appended via add reducer
}
The whole loop in one sentence

"The quality gate flags missing_areas → the planner turns those gaps into new sub-questions → the researcher chases only the new ids and appends sources via the reducer → control fields (research_iteration, researched_question_ids) keep it bounded and non-repeating." That's a self-correcting agent with no wasted work.

Nodes

Anatomy of a node

Every node has the same shape: a build_X(deps) factory returns an async def closure. The closure takes the state, does its work, emits start/finish events, and returns a partial dict.

def build_quality_check(deps: WorkflowDeps) -> NodeFn:
    model = deps.settings.model_for_node("quality_check")   # per-node model tier

    async def quality_check(state: GraphState) -> dict[str, Any]:
        async with emit_node("quality_check") as ev:   # brackets node_started / node_finished
            analysis = state.get("analysis")
            signals  = state.get("extracted_signals") or []
            user = quality_check_user(state["objective"], render_analysis(analysis), ...)

            result = await deps.llm.complete(
                QUALITY_CHECK_SYSTEM, user,
                response_model=QualityResult,        # structured output → validated
                model=model,
            )
            ev.set_preview({"coverage": result.coverage_score, ...})  # streamed to UI
            return {"quality": QualityVerdict(...), "retry_counts": ...}   # PARTIAL update

    return quality_check

Four recurring patterns to call out:

  • Factory + closure — build_X(deps) binds dependencies once; the returned function is what LangGraph calls. Keeps deps out of the state.
  • emit_node(...) — an async context manager that emits node_started on entry, node_finished (with a duration + preview) on clean exit, and node_errored on exception. This is what the live SSE stream renders.
  • Structured output — response_model=... forces the LLM into a validated Pydantic schema (with a JSON-mode + repair-retry fallback for models without native schema support).
  • Return a partial — never the whole state, just the keys this node produced.
The one node that does no LLM work

The researcher is pure I/O: it searches, fetches, and cleans pages under a bounded asyncio.Semaphore (fetch concurrency), with per-URL failures logged into errors and falling back to the search snippet. It only fails the node if it gathered nothing at all. Good example of "fan out concurrently, degrade gracefully."

Nodes · testability

Capabilities, not clients (the DI story)

Nodes never import a concrete OpenAI client or HTTP library. They close over a single WorkflowDeps bundle of protocols (interfaces). Production injects real clients; tests inject fakes — and the whole graph runs offline under pytest.

# backend/app/workflow/deps.py
@dataclass
class WorkflowDeps:
    llm: LLMClient            # protocol
    search: SearchClient      # protocol
    fetcher: Fetcher          # protocol
    settings: Settings

    @classmethod
    def from_settings(cls, settings) -> WorkflowDeps:
        return cls(
            llm=OpenRouterClient(settings),
            search=build_search_client(settings),
            fetcher=HttpxFetcher.from_settings(settings),
            settings=settings,
        )
Why this scores

It demonstrates dependency inversion: nodes depend on abstractions, not implementations. The payoff is concrete — the entire six-node graph runs in CI with zero network calls by passing fake llm/search/fetcher. And it dovetails with the serialization rule: deps live here, never in the checkpointed state.

Failure

Retries and failure handling

Two layers of failure handling: per-node retries for transient hiccups, and a run-level catch that records a typed failure (which a later resume can pick up from).

Node-level retry — transient only

The RetryPolicy only retries errors that might actually fix themselves. A schema/validation/empty-source error is deterministic — retrying just wastes tokens, so those fail fast.

# only transient transport errors are worth a retry
_TRANSIENT = (httpx.HTTPError, ConnectionError, TimeoutError, asyncio.TimeoutError)

def _retry_on(exc) -> bool:
    return isinstance(exc, _TRANSIENT)

Run-level — record a typed failure

The runner wraps the whole graph invocation; any uncaught error marks the run + session failed with a stable error code (e.g. cost_cap_exceeded) so the UI can show a meaningful reason — and resume stays possible.

# backend/app/workflow/runner.py (execute)
try:
    final_state = await self._invoke(run_id, session_id, resume=resume)
except Exception as exc:
    await self._fail(run_id, session_id, exc)   # mark failed, emit run_failed
    return
await self._complete(run_id, session_id, final_state)
What you'd add next (good to volunteer)

A second conditional edge for the research dead-end — if the researcher gathers zero sources, route to a degraded report node instead of failing the run. You sketched this; mentioning it shows you think about graph-level error paths, not just try/except.

Resume

Checkpointing & resume-from-crash

The reason this is fault-tolerant: LangGraph snapshots the entire state to Mongo after every super-step, via a custom MongoCheckpointer implementing BaseCheckpointSaver, keyed by thread_id == session_id.

# backend/app/workflow/runner.py (_invoke)
checkpointer = MongoCheckpointer(self.db)
graph = build_graph(self.deps, checkpointer=checkpointer)
config = {
    "configurable": {"thread_id": session_id},   # the checkpoint key
    "recursion_limit": self.settings.workflow_recursion_limit,
}
if resume:
    return await graph.ainvoke(None, config=config)   # None → continue from last checkpoint
return await graph.ainvoke(initial, config=config)   # fresh run from seeded inputs
The key trick: resume with None input

Calling ainvoke(None, config=...) with the same thread_id tells LangGraph to continue from the last checkpoint rather than start over. So a crash at the analyst resumes with plan, raw_sources, and extracted_signals already populated — completed nodes never re-run.

Why the super-step boundary?

LangGraph executes in super-steps (one "tick" where the eligible nodes run). Checkpointing at that boundary means state is always consistent — you never persist a half-updated state mid-node. Same instinct as committing a DB transaction at a clean boundary, not mid-write.

What's NOT guaranteed — say it first

Exactly-once external side effects. A node that crashed mid-LLM-call may re-issue that call on resume — so it's at-least-once, not exactly-once. Durable state ≠ exactly-once actions. That distinction is a genuinely senior thing to volunteer. (Graceful shutdown also flips a running session to a resumable interrupted on SIGTERM, so a deploy never strands a run.)

The live-streaming half of recoverability — reconnecting mid-run with no gaps or duplicates — is its own mechanism: see the SSE + sinceSeq deep dive.

"I needed a cyclic workflow with durable, resumable state — that's a state machine, not a chain. Gotchas

Subtleties an interviewer might probe

  • "Why is the routing function separate from the node?" — So the decision is a pure function of state, unit-testable without running the LLM node. The node produces the verdict; route_after_quality decides on it.
  • "What stops an infinite loop?" — Three things: the iteration < max_iterations guard, the empty-missing_areas guard, and LangGraph's recursion_limit backstop.
  • "Why de-dupe in both the researcher and the reducer?" — The researcher de-dupes within one pass (a question and a site crawl can hit the same URL); the reducer de-dupes across passes. Two different scopes.
  • "Where does seq / the DB handle live if not in state?" — In a RunContext carried via a contextvars.ContextVar, bound just before ainvoke. The state stays serializable; the live handles ride alongside.
  • "How do you test a node that calls an LLM?" — Inject a fake LLMClient via WorkflowDeps that returns canned structured output. No network, deterministic.

Flashcards

Rapid-fire: LangGraph workflow

What does a node return, and what does the framework do with it?

A node returns a partial dict — just the channels it produced, never the whole state. LangGraph merges it into the running state: plain channels are last-write-wins; channels annotated with a reducer combine old + new (e.g. raw_sources appends and de-dupes by URL).

Why does the quality loop go back to the planner, not the researcher?

Looping to the researcher would re-run it against the same plan — all question ids are already in researched_question_ids, so it gathers nothing new. The planner instead reads missing_areas and emits fresh gap-closing sub-questions, giving the next research pass new ground. (Commit a094b5a.)

What keeps the loop from running forever?

Three guards: research_iteration < max_iterations (a hard cap), missing_areas must be non-empty (otherwise the planner has no target and would just regenerate its original questions), and LangGraph's recursion_limit as a final backstop.

Why can't the DB client or event bus live in the state?

The whole state is checkpointed (JSON/msgpack) to Mongo after every super-step, so it must stay serializable — no live handles. Those live in a RunContext carried via a contextvars.ContextVar, bound just before ainvoke, visible to every node.

How does resume-from-crash actually work?

A custom MongoCheckpointer (BaseCheckpointSaver) snapshots state after every super-step, keyed by thread_id == session_id. To resume, the runner calls graph.ainvoke(None, config=...) with the same thread id — LangGraph continues from the last checkpoint, so completed nodes don't re-run.

Which errors get retried, and why not all of them?

Only transient transport errors (httpx.HTTPError, connection/timeout) via the node RetryPolicy. Deterministic failures — schema/validation/empty-source — won't fix on retry, so they fail fast instead of wasting tokens. An uncaught error fails the run with a typed code, leaving resume possible.

How is the graph testable without network access?

Nodes are closures over WorkflowDeps, which holds protocols (LLMClient, SearchClient, Fetcher). Tests pass fakes returning canned structured output, and compile the graph with no checkpointer — so the whole six-node graph runs deterministically offline in CI.

What's the honest limitation of the resume guarantee?

It guarantees durable state, not exactly-once side effects. A node that crashed mid-LLM-call may re-issue that external call on resume — so external actions are at-least-once. Distinguishing "durable state" from "exactly-once side effects" is the senior nuance.
