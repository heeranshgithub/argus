The Project

Argus in 60 seconds

Memorize this. It's your opening answer to "tell me about a project you're proud of."

The pitch

Argus is an AI research copilot for sales. You give it a target company; it runs a six-node LangGraph workflow that plans research questions, searches and fetches the web, extracts business signals, synthesizes them, and runs a quality gate that can loop back for more research. The output is a structured nine-section briefing with cited sources. You then chat with that briefing as grounded context — every answer cites its sources. You watch the whole thing execute live via server-sent events, and any run can resume from a crash because state is checkpointed to Mongo after every step.

The backend layering rule (say this verbatim)

The rule

api → services → repositories → db. Routes never touch Mongo directly. Repositories never raise HTTP. Domain errors raised in services get mapped to the wire error contract in one place (app/api/errors.py).

Why this matters in an interview: it shows you understand separation of concerns and dependency direction. Each layer has one job:

The three request flows

1 · Create a session — synchronous

POST /api/sessions {companyName, website, objective}
  → session_service.create_session → session_repo.create → Mongo
  ← 201 SessionOut

2 · Run the workflow — async + streamed

POST /api/sessions/{id}/run
  guard: session exists, no active run
  create run (status=running); BackgroundTasks.add_task(runner.execute)
  ← 202 { runId }            # returns immediately, work continues

GET .../runs/{runId}/events?since_seq=N   # SSE: backfill then tail

The key idea: the POST returns in milliseconds with a 202 Accepted; the minutes-long work runs in the background and the client watches it over a separate streaming connection. This is the async request-reply pattern — worth naming explicitly.

3 · Chat — grounded + streamed

POST /api/sessions/{id}/chat {content}
  require a report exists (else 409); load last 10 messages
  persist user msg + a "streaming" assistant msg
  asyncio.create_task(_generate)
  ← 200 { messageId }

_generate: BM25-select sources → build prompt (report + sources + history)

Four concepts to nail

Shared, typed state

Every node reads and writes one GraphState (a TypedDict): inputs (company_name, objective), accumulating outputs (plan, raw_sources, analysis, quality, report), and control fields (retry_counts, research_iteration, researched_question_ids).

Channels & reducers

Some state channels have a reducer — a merge function. raw_sources uses one that appends and de-dupes by URL across research loops, instead of clobbering. Plain channels are last-write-wins. This is the detail that shows you actually understand LangGraph, not just used it.

Conditional routing — and a subtle bug you fixed

quality_check returns needs_more_research + missing_areas. A conditional edge loops back — but back to the planner, not the researcher. Why does that distinction matter?

Why loop to the planner

Looping straight to the researcher would re-run it against the already-researched plan and gather nothing new — an infinite-ish loop burning tokens. Routing to the planner lets it convert the gaps into fresh sub-questions, so the next pass has something new to chase. (This is literally your commit a094b5a.) It's bounded by workflow_max_research_iterations and only taken when missing_areas is non-empty.

Capabilities, not clients

Nodes are closures over WorkflowDeps (llm, search, fetcher, settings) — and these are protocols (interfaces). So tests inject fakes and the whole graph runs offline under pytest. Crucially, deps never enter the state — state must stay JSON/msgpack-serializable for checkpointing, so no live DB handles or sockets allowed in it.

Failure handling

Per-node retry policy (workflow_node_retry_limit) for transient errors; an uncaught node error fails the run with a typed WorkflowError. You also sketched a second conditional edge for the "research dead-end" (zero sources found) that routes to a degraded report instead of crashing — good to mention as "what I'd add next."

The Project

Persistence & the naming bridge

MongoDB via Motor (the async driver). Five collection families: sessions, workflow_runs, reports, chat_messages, and the checkpointer collections. Indexes are created idempotently on startup (ensure_indexes).

Why document DB here (a likely question)

Why Mongo, not Postgres

The report and event shapes are evolving and nested (nine sections, arrays of signals with evidence URLs, an append-only event timeline). A document model fits a shape that's still changing without migrations, and Motor gives async access that matches the streaming workload. The honest counter: if the data were highly relational with strong cross-entity constraints, Postgres + JSONB would be the safer default. You chose flexibility for a single-tenant, evolving schema.

The camelCase ↔ snake_case bridge

One rule: camelCase on the wire, snake_case everywhere inside the backend and Mongo. A single base model does it:

class ApiModel(BaseModel):
    model_config = ConfigDict(
        alias_generator=to_camel,   # snake attr → camel alias
        populate_by_name=True,      # accept either on input
        from_attributes=True,

The intentional caveat

The bridge renames field keys only. Wire values that are enums or event-payload keys stay snake_case — event kind is run_started, a node name is signal_extractor. The frontend treats those as opaque string constants. (This is in your saved memory — interviewers love a candidate who knows exactly where their own abstraction stops.)

The Project · crown jewel

Live updates: SSE + sinceSeq

If you get one deep-dive question, hope it's this one. It's the most elegant part of the system and it touches networking, idempotency, and consistency.

The problem

A workflow run takes minutes and emits a stream of progress events. The user must see them live, and a page refresh or dropped connection mid-run must not lose or duplicate events. Same for chat token streaming.

Why SSE (not WebSocket, not polling)

OptionVerdict
SSE ✅One-way server→client is exactly our shape. Plain HTTP, native browser auto-reconnect, proxy-friendly, no framing to manage.
WebSocketBidirectional and powerful — but we have no client→server channel. It adds connection lifecycle, framing, and proxy complexity for zero benefit here.
PollingSimplest, but chatty and adds visible latency to a token stream. Wrong tool for real-time deltas.

The keystone: a monotonic sequence number

Every event carries a per-run seq. The durable record lives in workflow_runs.events in Mongo — Mongo is the source of truth. The stream endpoint takes ?since_seq=N and does three things in this order:

  1. Subscribe to the in-process event bus before reading backfill — this closes the race where an event lands between snapshot and subscription.
  2. Backfill stored events with seq > N.
  3. Tail live events, dropping anything ≤ N or already sent (dedupe by seq).
The payoff

A dropped connection reconnects with its latest seq → no gap, no duplicate. The chat stream mirrors this exactly with token-delta seqs, so a refresh mid-generation replays buffered deltas and tails the rest. This is the same idea as a Kafka consumer offset or TCP sequence numbers — name that analogy and you sound senior.

Own the limitation

The event bus and chat buffers are in-process. They don't span replicas — a multi-replica deploy needs Redis pub/sub. But because Mongo is the durable record, a reconnect still recovers correctly even after a restart. Stating this unprompted is a green flag.

Why subscribe-before-backfill order matters

This is a real concurrency subtlety. If you backfilled first and then subscribed, an event firing in that gap would be missed (not in your backfill snapshot, not yet in your subscription). Subscribing first means worst case you get a duplicate — which dedupe-by-seq harmlessly drops. Prefer at-least-once + idempotent dedupe over a lossy race.

The Project

Recoverability & the custom checkpointer

State is checkpointed to Mongo after every LangGraph super-step via a custom MongoCheckpointer implementing LangGraph's BaseCheckpointSaver, keyed by thread_id == session_id.

Two recovery paths

Resume after failure / interruption

A failed run leaves the session failed. POST /run/resume re-invokes the graph with None input, so LangGraph continues from the last checkpoint instead of re-running completed nodes. Graceful shutdown (SIGTERM) flips any running session to interrupted (a resumable terminal status) and cancels in-flight chat generation — so a deploy never strands a run.

Refresh / reconnect mid-run

The UI seeds from the persisted run snapshot, then attaches the SSE stream at the last seq. No work lost or redone.

What's NOT guaranteed (say it before they ask)

Exactly-once external side effects. A node that crashed mid-LLM-call may re-issue that call on resume — so it's at-least-once, not exactly-once. And cross-replica live streaming isn't there (in-process bus). Distinguishing "durable state" from "exactly-once side effects" is a genuinely senior distinction.

Why a super-step boundary?

LangGraph runs in super-steps (one "tick" of the graph where eligible nodes run). Checkpointing at that boundary means state is always consistent — you never persist a half-updated state mid-node. It's the same instinct as committing a DB transaction at a clean boundary, not mid-write.

Design Decisions

The five tradeoffs, as rehearsed answers

Each of these is a complete interview answer in the Context → Decision → Alternatives → Consequence shape. Practice saying them out loud.

D1 · OpenRouter as the single LLM gateway

Context: six nodes with different cost/quality needs (cheap planning vs. strong synthesis); must run real models without vendor lock-in.

Decision: route all LLM calls through OpenRouter via the OpenAI-compatible SDK. Models are config strings per node (llm_model_<node>), so each node pins an appropriate tier and a fallback chain handles provider outages.

Alternatives: direct OpenAI/Anthropic SDKs (best per-provider, but two SDKs/auth paths, no single per-node switch); LiteLLM proxy (similar, but another service to operate solo).

Consequence: one key, one client, per-node tiering, built-in fallback — at the cost of a hard dependency on OpenRouter uptime. Structured output handled defensively: native JSON-schema where supported, else JSON mode + schema-in-prompt + one repair retry.

D2 · SSE with sinceSeq, not WebSocket

Covered in depth in Live updates. One-liner: "One-way streams that must survive reconnects. SSE gives plain-HTTP resumability via a monotonic seq; WebSocket's bidirectional power is unused complexity here."

D3 · Custom Mongo checkpointer

Context: runs span minutes and several LLM calls; must resume from a crash; Mongo already runs everything.

Decision: implement BaseCheckpointSaver against Mongo, keyed by thread_id == session_id.

Alternatives: in-memory saver (zero setup, loses all on restart — fails the requirement); SQLite saver (durable, but a second datastore, awkward with async Motor); Redis saver (great for ephemeral, but not the system of record).

Consequence: one datastore, durable resume, checkpoints co-located with the run — at the cost of writing/maintaining the saver yourself.

D4 · BM25 over embeddings for chat retrieval

Zero extra deps, low latency, fully offline-testable. Gave up semantic recall — fine at ≤50 sources; at scale, swap to embeddings + a reranker.

D5 · In-process background tasks over a queue

No infra, instant start. Gave up horizontal scale and cross-process durability of the bus — though state stays durable in Mongo. Breaks down past ~5 concurrent runs.

The meta-point to land

Every one of these is a "right tool for the constraint of one engineer in a time-box, single-tenant" decision, each with a documented escape hatch. That story — "I optimized for shipping correctly under constraints, and I know exactly what to change when the constraints change" — is what senior interviewers are listening for.

Scaling Argus

"How would you scale this?" — the narrative

This is the classic follow-up. You've already written the answer as a debt list; here it is sequenced as a scaling story you can deliver top to bottom.

Step 0 · Name the current shape honestly

"It's deliberately single-worker, single-tenant. Background runs execute in the FastAPI worker; the event bus is in-process. That's correct for the demo's constraints and it's the first thing to change for scale."

The scaling ladder

BottleneckFixUnlocks
Runs block the API worker; ~5 concurrent maxMove runs to a real queue (Celery / Arq / RQ) with backpressure + retriesHorizontal scale of workers; API stays responsive
In-process event bus can't span replicasRedis pub/sub for the bus + chat buffersMultiple API replicas behind a load balancer
BM25 weak past ~50 sourcesEmbeddings + reranker (vector search)Semantic recall at scale
Single MongoRead replicas; later shard by tenant/session_idRead throughput, then write throughput
No auth / global sessionsMulti-tenant — sessions scoped to users, API keys/OAuthReal users, per-tenant isolation
Cost cap is soft, per-runHard pre-flight budget + token accountingPredictable spend under load
The shape of a great scaling answer

Always go identify the bottleneck → smallest change that removes it → what it now unlocks → the new bottleneck it exposes. Scaling is an iterative game of moving the bottleneck, not a single grand redesign. Saying that explicitly signals maturity.

Why a queue is the first move

Right now a run holds resources in the request worker's event loop. Under load that means event-loop contention and no backpressure — a burst of runs degrades everything, including fast read requests. A queue decouples submission from execution: the API just enqueues (fast) and workers drain at their own pace. This is the single most important distributed-systems pattern in the whole system — see Queues & async work.

The stateful catch with multi-replica + SSE

SSE connections are sticky to whichever replica holds the in-process bus that has the events. Two fixes, and you should mention both: (a) Redis pub/sub so any replica can serve any stream, or (b) sticky sessions at the LB as a stopgap. Because Mongo is the durable record, even a hard failover only costs a reconnect — the seq mechanism heals the gap.

System Design

A method for the open-ended question

"Design a URL shortener / news feed / rate limiter." There's a repeatable structure that keeps you from rambling.

  1. Clarify & scope — functional requirements, then non-functional (scale, latency, consistency, read/write ratio). Ask before you design. Nail down "how many users / QPS / data size."
  2. Back-of-envelope — estimate QPS and storage. Reads vs writes ratio drives everything (caching, replicas).
  3. API + data model — define the few core endpoints and the main entities/tables.
  4. High-level diagram — client → LB → app servers → cache → DB, plus async/queue paths.
  5. Drill into the hard part — they'll steer you (the hot key, the fan-out, the consistency). Go deep there.
  6. Bottlenecks & tradeoffs — name what breaks first and how you'd evolve it. End where you started Argus's debt list.

1 day ≈ 86,400s (~10⁵). So 1M requests/day ≈ ~12 req/s average, but plan for ~5–10× peak. Memory ns / SSD µs / network ms / disk-seek ms — each layer is ~1000× slower than the one above. Reads usually outnumber writes ~100:1 in consumer apps → cache aggressively.

System Design

Scaling: vertical, horizontal, stateless

Vertical vs horizontal

Vertical = bigger machine. Simple, but a hard ceiling and a single point of failure. Horizontal = more machines behind a load balancer. Near-unlimited, fault-tolerant, but requires your app to be stateless so any machine handles any request.

Why "stateless" is the magic word

If a server holds session state in memory, requests must return to that exact server — you can't freely add/remove machines. Push state to a shared store (Redis/DB) and your app tier becomes disposable and trivially scalable. This is exactly Argus's in-process-bus problem: the bus is local state that pins a stream to one replica. Naming that connection scores big.

Load balancing

Distributes traffic: round-robin, least-connections, or hashing. L4 (TCP) vs L7 (HTTP, can route by path/header). Also does health checks (pull dead nodes) and TLS termination. In Argus's deploy, nginx is the reverse proxy doing TLS termination in front of Uvicorn.

Sticky sessions

LB pins a client to one backend (by cookie/IP). Useful as a stopgap for in-memory state (like our SSE bus) but it undermines even load distribution and fault tolerance — prefer externalizing state instead.

System Design

Caching

The first lever for read-heavy systems. "Add a cache" is the most common right answer in design interviews — but know the failure modes.

Where caches live

Browser → CDN (edge) → reverse proxy → application cache (Redis/Memcached) → DB query cache. Each layer absorbs reads so the layer below sees fewer.

Patterns

PatternHowNote
Cache-aside (lazy)App checks cache; miss → read DB → populate cacheMost common. Stale risk; needs TTL/invalidation
Write-throughWrite to cache + DB togetherFresh reads; slower writes
Write-backWrite to cache, flush to DB asyncFast writes; data-loss risk on crash

Invalidation & the three classic failures

  • Stampede / thundering herd: a hot key expires and 10k requests hit the DB at once. Fix: lock/single-flight, or stagger TTLs.
  • Stale data: the cache and DB disagree. The hardest problem in CS ("there are only two hard things…"). TTLs bound staleness; explicit invalidation reduces it.
  • Hot key: one key gets disproportionate traffic. Fix: replicate the key, or add a local in-process tier.
Argus tie-in

Argus caches chat_suggestions on the session document — a tiny, deliberate application-level cache of an expensive-to-generate value. And RTK Query on the frontend is itself a client-side cache with tag-based invalidation.

Databases: SQL vs NoSQL, replication, sharding

SQL vs NoSQL — the real decision

SQL (Postgres/MySQL)NoSQL (Mongo/Dynamo)

StrengthJoins, transactions, constraintsHorizontal scale, evolving/nested docs
Scale modelScale up; sharding is harderBuilt to scale out
Pick whenRelational data, strong consistency needsDenormalized, high-volume, shape changing

Your Argus answer: "Evolving nested report/event shapes, single-tenant, async driver — Mongo fit. Relational integrity wasn't the constraint."

Replication

Leader-follower: one primary takes writes, replicates to read replicas. Scales reads and gives failover. The catch: replication lag → a follower may serve slightly stale data (read-after-write inconsistency). Mongo Atlas runs a replica set; for scale you'd send reads to secondaries.

Sharding (partitioning)

Split data across machines by a shard key to scale writes/storage. Choose the key to spread load evenly and keep common queries on one shard. Bad keys → hotspots (one shard overloaded). For Argus you'd shard by session_id or tenant. Cost: cross-shard queries and transactions get hard.

Indexes

An index trades write speed + storage for read speed. See the dedicated Indexes & B-trees section.

System Design

CAP, consistency, and ACID vs BASE

CAP theorem

In a network partition (P — unavoidable in distributed systems), you must choose between Consistency (every read sees the latest write) and Availability (every request gets a non-error response). You don't "pick 2 of 3" in the lazy sense — P is given, so it's really C vs A during a partition.

  • CP systems (e.g. a strongly-consistent DB) reject/​block during a partition to avoid serving stale data. Pick for money, inventory.
  • AP systems (e.g. Dynamo, Cassandra) keep serving and reconcile later. Pick for feeds, carts, likes.

Consistency models, weakest to strongest

Eventual (replicas converge "eventually") → read-your-writes → causal → strong/linearizable (acts like one copy). Stronger = more coordination = higher latency.

ACID vs BASE

ACID (Atomic, Consistent, Isolated, Durable) — transactional guarantees, classic RDBMS. BASE (Basically Available, Soft state, Eventual consistency) — the NoSQL/AP philosophy: trade strict consistency for availability and scale.

Argus tie-in

Argus is firmly consistency-tolerant where it can be: SSE replay tolerates at-least-once delivery and dedupes by seq; resume is at-least-once on external side effects. The durable invariant (Mongo as source of truth) is strong; the live layer is eventual. That layered choice is the sophisticated answer.

System Design

Queues & async work

The pattern behind Argus's #1 scaling fix, and behind half of all system-design answers.

What a queue buys you

  • Decoupling — producer (API) and consumer (worker) scale independently.
  • Backpressure / smoothing — a traffic spike fills the queue instead of crashing workers.
  • Resilience — a worker dies; the message is redelivered. Retries with backoff.
  • Async UX — return 202 Accepted immediately, do slow work later. (Argus already does this with BackgroundTasks — a queue is the durable, scalable version.)

Delivery semantics

At-most-once (may lose), at-least-once (may duplicate → consumers must be idempotent), exactly-once (expensive, often emulated via dedup + idempotency keys). Most real systems are at-least-once + idempotent consumers — exactly what Argus's seq-dedupe achieves at the stream layer.

Queue vs stream (log)

Task queue (RabbitMQ, SQS, Celery): a message is consumed once and gone. Event log (Kafka): an append-only, replayable log; consumers track their own offset and can rewind. Argus's workflow_runs.events + since_seq is a tiny event log — that's the exact analogy to draw.

Argus uses slowapi for per-IP limits on sensitive routes. In a distributed setup, the counter must live in Redis (shared), not per-instance memory — another instance of the stateless principle.

Idempotency

An operation is idempotent if doing it twice equals doing it once. Critical for retries and at-least-once delivery. Techniques: idempotency keys (client sends a UUID; server dedupes), natural idempotence (set-to-value vs increment), and dedupe by sequence number — which is precisely how Argus's SSE replay avoids double-rendering events.

CS Fundamentals

HTTP, TCP & the network stack

The journey of a request

DNS resolve → TCP handshake (SYN, SYN-ACK, ACK) → TLS handshake → HTTP request → response. Each round trip costs an RTT; this is why connection reuse (keep-alive, HTTP/2 multiplexing) and CDNs matter.

TCP vs UDP

TCP: connection-oriented, ordered, reliable (acks + retransmit), flow/congestion control. UDP: fire-and-forget, no ordering/delivery guarantee, low overhead — video, games, DNS.

HTTP methods & status codes you must know cold

Methods: GET (safe, idempotent), POST (create, not idempotent), PUT (replace, idempotent), PATCH (partial), DELETE (idempotent). Statuses: 200 OK, 201 Created, 202 Accepted (async!), 204 No Content, 301/302 redirect, 400 bad request, 401 unauth, 403 forbidden, 404, 409 conflict, 429 too many requests, 500, 502/503/504 gateway.

Argus tie-in

You use 202 for "run started, watch the stream," 409 for "chat with no report yet" and "run already active." Being able to justify a status-code choice is a small but real signal of API maturity.

HTTP/1.1 → /2 → /3

1.1: one request per connection at a time (head-of-line blocking). 2: multiplexed streams over one TCP connection. 3: over QUIC/UDP, removes TCP-level head-of-line blocking. SSE runs fine over any of them — it's just a long-lived HTTP response.

CS Fundamentals

Concurrency: async, event loops, threads

FastAPI + asyncio is core to Argus, and "what's the difference between concurrency and parallelism" is a perennial.

Concurrency ≠ parallelism

Concurrency: dealing with many things at once (interleaving). Parallelism: doing many things at once (multiple cores). A single-threaded event loop is concurrent, not parallel.

The async event loop

One thread runs a loop. On an await at an I/O point (network, DB), the coroutine yields control; the loop runs other ready coroutines until the I/O completes. Perfect for I/O-bound work (like Argus: mostly waiting on LLM/search/Mongo). Useless for CPU-bound work — a heavy computation blocks the whole loop.

The classic gotcha

A blocking call (sync DB driver, time.sleep, big CPU loop) inside an async handler freezes every other request on that worker. That's why Argus uses Motor (async Mongo) and the OpenAI async client — and why moving CPU/blocking work to a thread pool or a separate worker matters. It also explains your "~5 concurrent runs" ceiling: background runs share the event loop.

Processes vs threads vs coroutines

Processown memory; isolated; heavy; true parallelism
Threadshared memory; lighter; needs locks; GIL limits CPU parallelism in CPython
Coroutinecooperative; thousands cheap; one thread; great for I/O

Race conditions & coordination

Shared mutable state + concurrent access = races. Tools: locks/mutexes, semaphores, atomic ops, or avoiding shared state (message passing). Argus's "subscribe before backfill" ordering is a race-avoidance design — it makes a duplicate (harmless, dedupable) impossible to become a loss.

CS Fundamentals

Indexes, B-trees & query plans

Why an index is fast

Without one, a query scans every document (O(n)). An index is usually a B-tree — a balanced, high-fanout tree giving O(log n) lookups, range scans, and sorted reads. The cost: every write must also update the index (slower writes + storage).

Compound indexes & the prefix rule

Argus's chat_messages index is (session_id, created_at). Order matters: this index serves "messages for a session, sorted by time" in one shot — it filters by session_id then the tree is already ordered by created_at, so no separate sort step.

Why compound beats two single indexes

Two separate indexes force the DB to intersect results or pick one and re-sort. A compound index on (a, b) can satisfy filter-on-a-sort-by-b directly. The prefix rule: an index on (a,b) also serves queries on a alone, but not on b alone. Leftmost prefix wins.

Reading a query plan

Look for index scan (good) vs collection/table scan (bad at scale). explain() in Mongo / EXPLAIN ANALYZE in Postgres. Watch for in-memory sorts that should be index-ordered.

Flashcards

Rapid-fire: about Argus

Click to reveal. Practice answering before you open each one.

Walk me through what happens when a user clicks "Run."

POST returns 202 instantly after guarding (session exists, no active run) and registering a background task. The runner binds a RunContext (run id, seq counter, event bus, cost cap) and invokes the LangGraph with thread_id = session_id. Each node emits node_started → node_output* → node_finished events that are both appended to workflow_runs.events in Mongo and published to the in-process bus. The browser separately opens an SSE connection that backfills stored events past its since_seq then tails live ones. On success the run is marked completed and raw sources persisted (they fuel chat); on error, marked failed.

Why SSE and not WebSocket?

The data flow is one-way: server → client. SSE is plain HTTP with native auto-reconnect and a built-in Last-Event-ID that maps onto my sinceSeq resume. WebSocket gives a client→server channel I don't use, and adds connection lifecycle, framing, and proxy complexity. Polling would add latency to a token stream and be chatty.

How do you guarantee no lost or duplicate events on reconnect?

Every event has a per-run monotonic seq; Mongo's events array is the durable source of truth. On (re)connect with since_seq=N the server subscribes to the bus first, backfills seq > N, then tails live events dropping any ≤ N or already sent. Subscribe-before-backfill closes the race; dedupe-by-seq makes the worst case a harmless duplicate rather than a loss. Same mechanism for chat token deltas.

How does a crashed run resume without redoing work?

A custom MongoCheckpointer (LangGraph BaseCheckpointSaver) snapshots state after every super-step, keyed by thread_id = session_id. POST /run/resume re-invokes the graph with None input, so LangGraph continues from the last checkpoint. Caveat: external side effects are at-least-once — a node that died mid-LLM-call may re-issue it.

Why does the quality loop go back to the planner, not the researcher?

Looping to the researcher would re-run it against the same plan and gather nothing new. The planner converts the missing_areas into fresh sub-questions so the next pass chases new gaps. Bounded by a max-iterations cap and only taken when there are missing areas.

What breaks first under load, and how would you fix it?

Background runs execute in the FastAPI worker's event loop — past ~5 concurrent runs you get event-loop contention and no backpressure. First fix: move runs to a real queue (Celery/Arq/RQ) with backpressure and retries. That then exposes the in-process event bus as the next limit → Redis pub/sub so any replica can serve any SSE stream. Mongo stays the durable record throughout, so reconnects heal gaps.

How is the LLM grounded so chat answers cite sources?

On a chat message, I load the latest completed run's raw sources, BM25-select the most relevant, and build a prompt from a preamble + the report markdown + a numbered source pack + the last 10 messages. The model is instructed to cite with [i] markers; on completion I parse those markers back into structured citations stored on the message.

Flashcards

Rapid-fire: fundamentals

Concurrency vs parallelism?

Concurrency is structuring work so multiple tasks make progress by interleaving (one core can do it). Parallelism is literally executing multiple tasks simultaneously (needs multiple cores). An async event loop is concurrent but single-threaded.

CAP theorem in one breath?

During a network partition you must choose between consistency (every read sees the latest write) and availability (every request gets a response). Partitions are unavoidable, so it's really C-vs-A when one happens. CP for money/inventory, AP for feeds/carts.

How would you scale a read-heavy service?

Cache aggressively (cache-aside + TTLs), add read replicas, put a CDN in front of static/edge-cacheable content, and make the app tier stateless behind a load balancer so you can add machines. Then watch for hot keys and cache stampedes, and bound staleness with TTLs/invalidation.

Why must consumers of an at-least-once queue be idempotent?

Because a message can be redelivered (worker crash after processing but before ack, retries). If processing isn't idempotent, you double-charge / double-send. Use idempotency keys, dedupe by id, or naturally idempotent operations (set vs increment).

Why is a compound index (a, b) better than two indexes here?

It serves filter-on-a-then-sort-by-b in a single ordered traversal — no result intersection, no separate sort. By the leftmost-prefix rule it also serves queries on a alone. Two single indexes would force an intersection or a re-sort.

What makes a service horizontally scalable?

Statelessness. If no request depends on in-memory state local to one server, any server can handle any request, so you can add/remove machines freely behind a load balancer. Push state to a shared store (Redis/DB). The opposite — like an in-process event bus — pins clients to one node.
