Live streaming with SSE + sinceSeq

How Argus streams a minutes-long workflow (and token-by-token chat) to the browser so that a refresh or a dropped connection never loses an event and never shows a duplicate. We'll build the idea up, then read the real code line by line.

Start here

The problem, stated precisely

A workflow run takes minutes and emits a stream of progress events (node started, output, finished…). The UI must show them live. But networks are hostile: the user refreshes the tab, the wifi blinks, a phone sleeps. The hard requirement:

The invariant we must hold

After any refresh, reconnect, or dropped connection mid-run, the client ends up with exactly the set of events that happened — no gaps, no duplicates — and keeps receiving new ones live. The same must hold for chat replies streamed token-by-token.

That's deceptively hard. Naive live streaming loses whatever fired while you were disconnected. Naive "replay everything on reconnect" duplicates what you already saw. We need a mechanism that does both correctly, and Argus's answer is small and elegant: a durable event log + a per-run sequence number + idempotent dedupe.

The core idea

The whole trick in one picture

Forget code for a second. The entire design is three moving parts:

  1. A durable, append-only log. Every event is written to Mongo (workflow_runs.events) the moment it happens. Mongo is the source of truth — if everything else fails, the full history is here.
  2. A monotonic sequence number (seq) on every event. 1, 2, 3, … per run. This is the event's permanent address.
  3. A live bus + a resume cursor. Connected clients get events pushed live. On (re)connect, the client says "I last saw seq = N" via ?since_seq=N, and the server replays everything after N, then tails live.
  events happen →  seq:  1    2    3    4    5    6 ...
                        ───────────────●───────────────▶ time
  client saw up to seq=3 ─────────────┘
  reconnects with ?since_seq=3
                        →  server replays 4,5,6  then streams 7,8,...
                           (anything ≤ 3 is dropped — already seen)
You already know this pattern

This is exactly a Kafka consumer offset and, one layer down, how TCP sequence numbers recover a byte stream after loss. A durable ordered log + a per-consumer cursor is the canonical way to get exactly-once delivery semantics out of an at-least-once channel. Naming that analogy in an interview is a senior signal.

Concept

What SSE actually is (on the wire)

Server-Sent Events is not a fancy protocol — it's one long-lived HTTP response with a special content type that stays open and dribbles out text frames. That simplicity is the whole point.

The server responds with Content-Type: text/event-stream and never closes the body. Each event is a few lines followed by a blank line:

id: 4
event: node_finished
data: {"runId":"run_789","node":"researcher","seq":4,...}

id: 5
event: node_started
data: {"runId":"run_789","node":"analyst","seq":5,...}

Three fields matter:

FieldMeaning
id:The event id. The browser remembers the last one and re-sends it as the Last-Event-ID header on auto-reconnect. Argus puts the seq here.
event:A named event type. The client listens per-name (node_started, delta, …).
data:The payload — Argus sends camelCase JSON.
Why SSE here and not WebSocket or polling

Our data flow is strictly one-way, server→client. SSE is plain HTTP (works through every proxy, no upgrade handshake), has built-in auto-reconnect, and needs no framing protocol. WebSocket is bidirectional — it would hand us a client→server channel we never use, plus connection-lifecycle and proxy complexity. Polling is chatty and adds latency to a token stream. SSE fits the shape exactly.

Nuance worth saying out loud

SSE gives you Last-Event-ID for free, but Argus deliberately drives reconnection itself and passes the cursor as an explicit ?since_seq= query param instead. Why? Control. Managing reconnect in our own code lets us add exponential backoff, offline awareness, and one unified resume mechanism shared with the chat stream — rather than depending on each browser's EventSource retry behaviour.

Concept

The three guarantees, and how seq delivers them

Everything reduces to one integer per event. Watch how a single monotonic seq buys all three properties:

GuaranteeMechanism
No lossEvery event is durably appended to Mongo before we care about live delivery. On reconnect, backfill replays everything with seq > since_seq from that durable log. Even events dropped by an overloaded live queue are recovered here.
No duplicatesThe client tracks the highest seq it has seen. The server drops any event ≤ since_seq, and within a connection keeps a sent set to skip anything already delivered.
ResumabilityThe cursor is the client's last seq. Reconnect = "give me everything after N." No server-side per-client session to maintain.
The mental compression

seq is the event's permanent address; since_seq is the client's bookmark. Loss-free + dup-free streaming is just "replay from the bookmark, then follow along, ignoring anything at or before it."

Concept

At-least-once + idempotent dedupe

A foundational distributed-systems idea that this design embodies — be ready to say it.

Delivery guarantees come in three flavours:

  • At-most-once — never duplicates, but may lose. (Fire and forget.)
  • At-least-once — never loses, but may duplicate. (Retry until acked.)
  • Exactly-once — the dream; genuinely hard, usually emulated.
How Argus "fakes" exactly-once

The transport is deliberately at-least-once: on the race window (next section) it would rather send a duplicate than risk a gap. Then it makes duplicates harmless by deduping on seq — an idempotent consumer. At-least-once delivery + an idempotent consumer = effectively exactly-once processing. That's the same recipe behind idempotency keys in payment APIs and Kafka's exactly-once semantics. Choosing "duplicate-but-safe" over "lossy-but-clean" is the heart of the design.

Backend code

① The event, and its two sinks

An event is a plain dict. Notice seq is a first-class field, and the same shape is both the durable record and the SSE payload.

# backend/app/workflow/events.py
class WorkflowEvent(TypedDict):
    """One entry in a run's timeline (also the SSE payload shape)."""
    run_id: str
    session_id: str
    node: str
    kind: str          # run_started, node_finished, …
    payload: dict[str, Any]
    ts: datetime
    seq: int           # ← the permanent address

Every event flows to two sinks in emit() — first Mongo (durable), then the live bus (best-effort):

async def emit(kind, *, node=RUN_NODE, payload=None, node_status=None):
    ctx = current_run_context()
    event: WorkflowEvent = {
        "run_id": ctx.run_id,
        "session_id": ctx.session_id,
        "node": node, "kind": str(kind), "payload": payload or {},
        "ts": _now(),
        "seq": ctx.next_seq(),          # ← assign the next seq
    }
    await workflow_repo.append_event(ctx.db, ctx.run_id, event, node_status=node_status)
    if ctx.bus is not None:
        ctx.bus.publish(ctx.session_id, event)   # best-effort live fan-out
    return event
Order matters: Mongo first, bus second

The durable write happens before the live publish. So an event that exists for any live subscriber is guaranteed to already be in the source of truth. A failed or skipped publish never corrupts the record — a reconnect always recovers it via backfill. This ordering is the foundation of "no loss."

Where seq comes from: the RunContext

The counter lives in a per-run context object carried via a contextvars.ContextVar — deliberately not in GraphState, because the graph state is checkpointed to Mongo and must stay serializable (no live handles/counters).

@dataclass
class RunContext:
    db: AsyncIOMotorDatabase
    run_id: str
    session_id: str
    bus: EventBus | None = None
    _seq: int = field(default=0)

    def next_seq(self) -> int:
        self._seq += 1            # monotonic, per-run
        return self._seq

And the durable append is a single atomic Mongo update — a $push onto the events array, optionally updating the node's status in the same write:

# backend/app/repositories/workflow_repo.py
async def append_event(db, run_id, event, *, node_status=None):
    update = {"$push": {"events": dict(event)}}
    if node_status is not None and event["node"]:
        update["$set"] = {f"node_status.{event['node']}": node_status.value}
    await db[WORKFLOW_RUNS].update_one({"_id": _to_object_id(run_id)}, update)
Backend code

② The in-process event bus

The live side is a tiny pub/sub: each connected SSE handler subscribes a queue for its session_id; emit() publishes to every queue for that session. If nobody's listening, publish is a no-op — the event still lives in Mongo.

class EventBus:
    def __init__(self, *, max_queue=512):
        self._subscribers: dict[str, set[asyncio.Queue]] = {}
        self._max_queue = max_queue
        self.dropped_events = 0

    def subscribe(self, session_id):
        queue = asyncio.Queue(maxsize=self._max_queue)
        self._subscribers.setdefault(session_id, set()).add(queue)
        return queue

    def publish(self, session_id, event):
        for queue in self._subscribers.get(session_id, set()):
            if queue.full():
                with contextlib.suppress(asyncio.QueueEmpty):
                    queue.get_nowait()          # evict OLDEST to make room
                self.dropped_events += 1
            queue.put_nowait(event)
Why bounded queues + drop-oldest

A slow or stalled browser must never block the workflow. So each subscriber queue is bounded (512). On overflow we evict the oldest event — live consumers always trend toward the newest. Dropped events aren't lost: they're still in Mongo, so a reconnect with since_seq recovers them. This is backpressure handling: protect the producer, degrade the slow consumer gracefully, lean on the durable log for correctness.

Backend code · the crown jewel

③ The stream endpoint: subscribe → backfill → tail

This is the function to understand cold. Read it slowly — every line defends the invariant. The ordering of the first two steps is the whole game.

# backend/app/api/runs.py — GET /sessions/{id}/runs/{runId}/events?since_seq=N
async def generator():
    queue = event_bus.subscribe(session_id)   # ① SUBSCRIBE FIRST (before backfill)
    sent: set[int] = set()
    try:
        # ② BACKFILL: replay durable events the client hasn't seen
        for event in run.events:
            if event.seq <= since_seq:
                continue                       # already seen → skip
            sent.add(event.seq)
            yield _sse_format(frame)

        # if the run already finished, the terminal event is in the backfill —
        # close now instead of waiting on a queue that'll never fill
        if run.status in _TERMINAL_STATUSES:
            return

        # ③ TAIL: stream live events until the run terminates
        while True:
            if await request.is_disconnected():
                return                          # client gone → don't leak a subscriber
            try:
                live = await asyncio.wait_for(queue.get(), timeout=15.0)
            except TimeoutError:
                yield ": keep-alive\n\n"          # heartbeat (SSE comment)
                continue
            if (live["run_id"] != run_id
                    or live["seq"] <= since_seq      # DEDUPE: below the cursor
                    or live["seq"] in sent):       # DEDUPE: already sent (the race!)
                continue
            sent.add(live["seq"])
            yield _sse_format(live)
            if live["kind"] in _TERMINAL_KINDS:
                return                          # run_completed / run_failed → close
    finally:
        event_bus.unsubscribe(session_id, queue)   # always clean up

Walk the four jobs it does:

  1. Subscribe first. Register the live queue before reading the stored history. (Why this exact order is the next section.)
  2. Backfill. Replay durable events with seq > since_seq, recording each in sent.
  3. Short-circuit finished runs. If the run already ended, the terminal event was in the backfill — return immediately rather than block on a queue that will never receive anything.
  4. Tail. Pull live events; skip wrong-run, below-cursor, and already-sent ones; emit a heartbeat every 15s of silence; close on the terminal event or client disconnect; and always unsubscribe in finally.

The SSE frame format — note seq becomes the SSE id::

def _sse_format(event):
    data = WorkflowEventOut.model_validate(event).model_dump(by_alias=True, mode="json")
    return f"id: {event['seq']}\nevent: {event['kind']}\ndata: {json.dumps(data)}\n\n"
The subtlety

④ Why subscribe-before-backfill (the race)

This is the single most interview-worthy line in the system. It's a genuine concurrency race, and the fix is one line of ordering.

Between reading the stored history and starting to listen live, time passes. An event can fire in that gap. Consider the wrong order:

WRONG: backfill, then subscribe
1. read stored events (snapshot up to seq 5)
↓
⚡ seq 6 fires herenot in your snapshot, no subscriber yet
↓
2. subscribe — too late
↓
seq 6 is LOST
RIGHT: subscribe, then backfill
1. subscribe (now capturing live)
↓
⚡ seq 6 fires herequeued live AND about to be in backfill
↓
2. backfill replays … 6
↓
seq 6 arrives twicededupe by sent drops the copy
The principle

By subscribing first, the worst case becomes a duplicate (seq 6 in both the backfill and the live queue) — and the sent set makes duplicates free. The wrong order makes the worst case a loss, which is unrecoverable on that connection. You deliberately trade a possible-but-harmless duplicate to make a loss impossible. That's the at-least-once + idempotent-dedupe principle applied to a single race window.

Backend code

⑤ The chat stream mirrors it exactly

Token-by-token chat replies use the same shape, with an in-memory ChatStreamManager instead of Mongo as the buffer (a reply is short-lived, so its deltas are buffered in process and replayed on reconnect). Each delta gets its own seq:

# backend/app/services/chat_service.py
class ChatStreamManager:
    def publish_delta(self, message_id, text):
        state = self._streams.get(message_id)
        state._seq += 1
        seq = state._seq
        state.deltas.append((seq, text))     # BUFFER for replay on reconnect
        frame = {"type": "delta", "seq": seq, "text": text}
        for queue in list(state.subscribers):
            queue.put_nowait(frame)           # live fan-out

The SSE handler for chat does the same buffered-backfill (seq > since_seq) then tail dance, so a refresh mid-generation replays the tokens you already saw being typed and continues from there. The stream ends on a done frame (carrying final citations + status) rather than waiting for more.

Same idea, different durability

Workflow events are durable in Mongo (runs are long, audit matters). Chat deltas live in an in-process buffer (a reply finishes in seconds). But the protocol is identical: monotonic seq, replay-past-cursor, dedupe, tail. One mental model covers both feeds.

Frontend code

⑥ The browser side: track seq, reconnect with it

The client's job is tiny: remember the highest seq seen, and reconnect carrying it. Argus wraps EventSource and drives reconnection manually for control.

// frontend/src/lib/sse.ts
let lastSeq = opts.sinceSeq ?? 0;

const url = () => {
  const u = new URL(baseUrl);
  u.searchParams.set("since_seq", String(lastSeq));   // carry the cursor
  return u.toString();
};

const handle = (ev) => {
  const event = JSON.parse(ev.data);
  if (typeof event.seq === "number")
    lastSeq = Math.max(lastSeq, event.seq);   // advance the bookmark
  opts.onEvent(event);
};

const connect = () => {
  if (navigator.onLine === false) {            // offline-aware: don't hammer
    timer = setTimeout(connect, 2000); return;
  }
  source = new EventSource(url());            // reconnect with latest since_seq
  source.onerror = () => {
    source?.close();
    const delay = Math.min(1000 * 2 ** retry, 10_000);  // 1s→2s→4s… cap 10s
    retry += 1;
    timer = setTimeout(connect, delay);       // exponential backoff
  };
};
What the frontend does NOT have to do

No reconciliation logic, no "did I miss anything?" bookkeeping, no client-side dedup beyond letting the server filter. The cursor (lastSeq) plus the server's backfill+dedupe make the client almost stateless about correctness. Push the hard guarantee to one place (the server + the durable log); keep the client dumb. That's good system design.

On reconnect the loop just calls connect() again with the updated lastSeq — the server replays the gap and resumes the live tail. The user sees a brief pause, never a gap or a double-rendered node.

Operational detail

Heartbeats & surviving proxies

Two real-world gotchas that the code quietly handles:

Heartbeats

If no event fires for a while, idle connections get reaped by proxies/load balancers. So after 15s of silence the server emits an SSE comment line (: keep-alive\n\n) — ignored by the client, but it keeps the TCP connection and any intermediary awake.

Proxy-defeating headers

_SSE_HEADERS = {
    "Cache-Control": "no-cache",        # don't cache a stream
    "Connection": "keep-alive",
    "X-Accel-Buffering": "no",         # tell nginx: do NOT buffer — flush each frame
}
The classic SSE bug

Without X-Accel-Buffering: no, nginx (Argus's reverse proxy) buffers the response and the user sees nothing for minutes, then everything at once when the buffer flushes — the stream "works" but isn't live. Knowing this header exists is a strong practical signal.

Own the tradeoff

The limitation: in-process bus

State this before they ask — it's the honest edge of the design.

What breaks at scale

The EventBus and chat buffers live in one process's memory. With multiple API replicas behind a load balancer, a client's SSE connection might land on a replica that isn't the one running its workflow — so it sees no live events. The fix is Redis pub/sub (or similar): publish events to a shared channel any replica can subscribe to.

But the design degrades gracefully

Because Mongo is the durable source of truth and the resume protocol is since_seq, even today a process restart or failover only costs the client a reconnect — the backfill heals the gap. The in-process bus is a performance/liveness limitation, not a correctness one. That separation — "durable correctness vs. in-memory liveness" — is exactly the nuance senior interviewers listen for.

Other honest edges: exactly-once is only at the delivery layer — a workflow node that crashed mid-LLM-call may re-issue that external call on resume (at-least-once side effects). The seq mechanism guarantees the event stream is clean, not that every external action ran exactly once.

Flashcards

Rapid-fire: SSE + sinceSeq

Why SSE and not WebSocket?

The flow is one-way server→client. SSE is plain HTTP with built-in auto-reconnect and no framing, and works through proxies untouched. WebSocket adds a client→server channel we don't use, plus connection lifecycle and proxy complexity. Polling would be chatty and add latency to a token stream.

How do you guarantee no lost and no duplicate events on reconnect?

Each event has a monotonic per-run seq and is durably stored in Mongo before live fan-out. On reconnect the client sends ?since_seq=N; the server subscribes to the live bus first, backfills stored events with seq > N, then tails live, dropping anything ≤ N or already in its sent set. No gap (backfill from the durable log), no dup (dedupe by seq).

Why subscribe to the bus before reading backfill?

To close a race: an event can fire between snapshotting the stored history and starting to listen. Subscribe-first means that event is captured live (and may also be in the backfill) → worst case a duplicate, which dedupe-by-seq drops harmlessly. Backfill-first would lose it. You trade a possible duplicate to make a loss impossible.

What happens if a client is too slow to consume?

Each subscriber queue is bounded (512). On overflow the bus evicts the oldest event and bumps a counter — it never blocks the workflow producer. The dropped events remain in Mongo, so when the client catches up or reconnects with since_seq, backfill recovers them. Backpressure handled; correctness preserved by the durable log.

Why isn't seq stored in the graph state?

The LangGraph state is checkpointed to Mongo after every super-step and must stay serializable — no live handles or counters. So the seq counter, DB handle, and bus live in a RunContext carried via a contextvars.ContextVar, bound just before the graph is invoked and visible to every node.

What's the scaling limitation and is it a correctness problem?

The event bus and chat buffers are in-process, so they don't span multiple API replicas — a fix is Redis pub/sub. But it's a liveness limitation, not correctness: Mongo is the durable source of truth, so after a failover or restart a client just reconnects and backfill heals the gap.

How does the chat token stream differ from the workflow stream?

Same protocol — monotonic seq, backfill-past-cursor, tail, dedupe — but the buffer is in-process (ChatStreamManager) rather than Mongo, because a reply is short-lived. Deltas are buffered for replay on mid-generation refresh; the stream ends on a done frame carrying final citations and status.
