Live data, two halves of one trick

The chat reply feels instant and arrives token-by-token because two patterns work together: Optimistic UI paints the message before the server confirms it, and Server-Sent Events (SSE) stream the actual tokens in afterward. This page takes both apart — concept first, then the exact Argus code — until you can rebuild them from memory and defend every line.

The one-sentence mental model

"Optimistic UI is a promise the UI makes; SSE is how the server keeps it; reconciliation is how we check the promise was kept. The optimistic placeholder fills the gap between the click and the first token; the stream fills in the answer; the refetch replaces guesses with persisted truth."

Why this combination is a senior signal

  • Most candidates know one of these. Pairing them — and naming the failure mode of each — is what reads as experience.
  • It forces you to talk about state ownership: what's the source of truth (server cache), what's ephemeral (live tokens), what's a temporary lie (the optimistic placeholder).
  • It naturally leads into resilience: reconnection, resume cursors, dedup, cleanup — all the things interviewers probe after the happy path.
SSE

What SSE actually is

Server-Sent Events is a browser standard for a one-way, server→client, long-lived stream of text messages over a single ordinary HTTP response. The server holds the response open and keeps writing; the browser fires an event for each message. That's the entire idea.

How it differs from the alternatives

ApproachDirectionConnectionGood for
Pollingclient asks repeatedlymany short requestssimple, but laggy + wasteful
Long-pollingclient asks, server holdsrequest per message-ishokay fallback, awkward

"Why SSE and not WebSockets?" — the answer that wins

"My data is one-directional: the server pushes workflow progress and LLM tokens; the client never needs to push back over the same channel (user actions go through normal REST POSTs). SSE gives me exactly that over plain HTTP — so it rides through proxies, load balancers, and HTTP/2 with no protocol upgrade, and the browser's EventSource gives me automatic reconnection and an event id / resume mechanism for free. WebSockets are full-duplex and heavier; I'd reach for them only if the client needed to stream to the server too. Picking the lighter tool that fits the data flow is the point."

The constraints to know

SSE is text only (UTF-8 — send JSON as a string, not binary). Over HTTP/1.1 a browser allows only ~6 connections per domain, so many open streams can starve other requests — HTTP/2 multiplexing removes that limit (Argus runs behind HTTP/2). And it's server→client only by design.

SSE

The wire format (this demystifies everything)

People treat SSE as magic until they see the bytes. It's a dead-simple text protocol. The server responds with Content-Type: text/event-stream and then writes UTF-8 text frames. Each frame is a few field: value lines, and a blank line terminates the frame and dispatches the event.

HTTP/1.1 200 OK
Content-Type: text/event-stream      // ← this header is what makes it SSE
Cache-Control: no-cache
Connection: keep-alive

event: delta                          // the event NAME (default is "message")
id: 1                                 // becomes lastEventId — the resume cursor
data: {"seq":1,"text":"Based"}        // the payload (we send JSON as a string)
                                      // ↑ BLANK LINE = end of frame, fire the event
event: delta
id: 2
data: {"seq":2,"text":" on the"}

event: done                           // a differently-named event ends the reply
id: 3
data: {"messageId":"m_42","status":"complete","citations":[...]}

: this is a comment / heartbeat — keeps the connection warm
FieldMeaning
event:The event name you listen for. Omitted ⇒ the generic "message" event. Argus uses named events (delta, done, node_started…).
data:The payload line(s). Multiple data: lines are joined with \n. We put a JSON string here and JSON.parse it on arrival.
id:An event id. The browser remembers the last one and sends it back as the Last-Event-ID header on auto-reconnect — the native resume mechanism.
retry:Milliseconds the browser should wait before auto-reconnecting. (Argus drives its own reconnect instead — see below.)
: (colon)A comment line. Used as a heartbeat to stop idle proxies from killing the connection.
Say this and you sound like you've shipped it

"SSE isn't a special socket — it's a normal HTTP response the server never closes, streaming text/event-stream frames. Each frame is event:/data:/id: lines ended by a blank line. That simplicity is why it survives proxies and CDNs that mangle WebSocket upgrades."

SSE

The client API: EventSource

The browser ships a built-in client for this protocol: EventSource. You give it a URL; it opens the stream, parses frames for you, and dispatches DOM events. You never touch the raw text.

const es = new EventSource(url);          // opens the GET stream immediately

es.onopen    = () => {/* connected */};
es.onerror   = () => {/* dropped — by default it auto-retries */};

// generic "message" events (frames with no `event:` line):
es.onmessage = (ev) => console.log(ev.data);

// NAMED events need explicit listeners — this is the key gotcha:
es.addEventListener("delta", (ev) => {
  const { seq, text } = JSON.parse(ev.data);  // data is always a string
});
es.addEventListener("done", (ev) => { /* … */ es.close(); });

es.close();                            // you MUST close it — or it reconnects forever
The two traps everyone hits

(1) Named events bypass onmessage. If the server sends event: delta, your onmessage handler will never fire — you must addEventListener("delta", …). This is exactly why Argus registers one listener per event kind. (2) EventSource auto-reconnects on its own unless you close() it. Forget the cleanup and a "done" stream silently reopens, replaying or double-handling — which is why every Argus stream returns an explicit teardown.

Why Argus wraps it instead of using it raw

Native EventSource auto-reconnect re-sends Last-Event-ID, but we want the cursor in the URL (?since_seq=) so the backend's replay logic is dead simple and testable, and we want our own backoff plus an offline check. So Argus disables the implicit retry path (it close()s on error and reconnects manually). That trade — give up the freebie, gain control of the resume contract — is a great thing to be able to justify.

SSE

Resumability: the since_seq contract

This is the part that turns a toy stream into a production one. Networks drop. Tabs sleep. Users refresh mid-generation. The question is: after a gap, how do you replay exactly what was missed — no gap, no duplicate?

The idea: every event carries a monotonic seq

The backend stamps each event with an ever-increasing sequence number and persists them. The client tracks the highest seq it has seen. On (re)connect it asks for ?since_seq=N, and the server replays only events with seq > N.

Client has seen up to seq 7
↓
reconnect with ?since_seq=7
↓
No gap, no dupe8,9,10 delivered; 1–7 never resent

Here's the exact cursor-tracking from lib/sse.ts — note the cursor is updated on every frame and folded into the reconnect URL:

// lib/sse.ts — the resume cursor lives in a closure variable
let lastSeq = opts.sinceSeq ?? 0;

const url = () => {
  const u = new URL(baseUrl);
  u.searchParams.set("since_seq", String(lastSeq));  // ← cursor rides in the query string
  return u.toString();
};

const handle = (ev) => {
  const event = JSON.parse(ev.data);
  if (typeof event.seq === "number")
    lastSeq = Math.max(lastSeq, event.seq);          // advance the high-water mark
  opts.onEvent(event);
};

The reconnect loop: exponential backoff + an offline gate

// lib/sse.ts — we drive reconnection ourselves
const connect = () => {
  if (closed) return;                              // torn down → stop
  if (navigator.onLine === false) {                // don't hammer a dead network
    timer = setTimeout(connect, 2000); return;
  }
  source = new EventSource(url());                  // url() embeds the latest since_seq
  source.onopen = () => { retry = 0; opts.onOpen?.(); };  // reset backoff on success
  for (const kind of EVENT_KINDS)
    source.addEventListener(kind, handle);            // one listener per NAMED event
  source.onerror = () => {
    source?.close(); source = null;
    const delay = Math.min(1000 * 2 ** retry++, 10_000);  // 1s,2s,4s… cap 10s
    timer = setTimeout(connect, delay);
  };
};
Why exponential backoff (and why cap it)

"If the server is down, reconnecting every 100ms is a self-inflicted DDoS that delays its recovery. Exponential backoff (1→2→4→8s) backs off fast so a momentary blip retries quickly but a real outage doesn't flood. I cap at 10s so the client still recovers promptly once the server returns, and I reset the counter on a successful open so the next blip starts fresh. The offline check is a cheaper guard — if navigator.onLine is false there's no point even trying."

The line that proves you get it

"Resumability needs three things to line up: the server must persist + sequence events, the client must track the high-water mark, and the reconnect must carry that cursor so the server replays the tail. Miss any one and you get gaps or duplicates. The seq is also what makes my client-side merge idempotent — same number, same dedupe key."

Argus

The wrapper, end to end

Argus has two SSE consumers sharing the same shape: openEventStream (workflow progress, many named events) and openChatStream (a chat reply: delta ×N then done). Both return an unsubscribe function — the single most important detail for React.

SSR-safe by construction

export function openEventStream(baseUrl, opts) {
  if (typeof window === "undefined" || typeof EventSource === "undefined") {
    return () => {};      // no-op on the server — EventSource is browser-only
  }
  // … browser path …
  return () => { closed = true; clearTimeout(timer); source?.close(); };
}

Because Next.js renders components on the server first, a hook that opened a stream during SSR would crash (EventSource doesn't exist in Node). Returning a no-op teardown means the calling useEffect still gets a valid cleanup function and the whole thing is safe either side of hydration.

Chat vs run: the only real difference is the terminator

// openChatStream — 'done' is special: it terminates, it must not reconnect
const onDone = (ev) => {
  opts.onDone(JSON.parse(ev.data));
  close();   // ← sets closed=true so onerror's reconnect path is a no-op
};
Subtle bug this avoids

If done didn't set closed = true, the server closing the connection right after the final frame would look like an error to EventSource, trigger onerror, and reconnect a finished stream — re-streaming the whole reply. The closed flag makes the terminator authoritative: once done lands, every later callback short-circuits.

Optimistic UI

What it is, and the risk

Optimistic UI updates the screen assuming the operation will succeed, before the server has confirmed it — so the app feels instant. You render the expected result now, fire the request in the background, and reconcile when the truth arrives (or roll back if it failed).

Pessimistic (naive)click → spinner → wait → render
↓
feels slow, even if it's correct
Optimisticclick → render guess NOW → reconcile later
↓
feels fast — if you handle the failure path
The interview answer (the one already in the playbook, expanded)

"Optimistic UI updates the screen assuming the operation will succeed, before the server responds, so the app feels instant. The risk is divergence if it fails — so you need a reconciliation or rollback path. In Argus, when you send a chat message I immediately render an optimistic user+assistant placeholder pair and start streaming tokens into the assistant one; once the real persisted rows arrive via refetch I drop the placeholders, and a failed reply surfaces a toast and is marked failed. So the optimistic state always converges with the server truth — it's never left lying."

The three things every optimistic update needs

  1. An immediate local write — render the predicted result (the placeholder pair).
  2. A reconciliation path — when the server's truth lands, replace the guess with it (the refetch + overlay).
  3. A failure path — if it fails, roll back or mark it (the toast + status: "failed"). Never leave a guess on screen as if it were real.
Argus

The send() flow, line by line

Here's the real send from use-chat-stream.ts. Four things happen in deliberate order:

// hooks/use-chat-stream.ts
const send = useCallback(async (content) => {
  try {
    // 1. POST the message. The server creates the row + an empty assistant
    //    row and returns the assistant messageId we'll stream into.
    const { messageId } = await postChat({ sessionId, content }).unwrap();

    // 2. OPTIMISTIC: paint a placeholder user+assistant pair right now,
    //    keyed to the real assistant id so the stream can target it.
    setPending({
      userId: `pending-user-${messageId}`,
      userContent: content,
      assistantId: messageId,
      createdAt: new Date().toISOString(),
    });

    // 3. STREAM: open the SSE reply from seq 0 — tokens render immediately,
    //    without waiting on the history refetch.
    beginStream(messageId, 0);

    // 4. RECONCILE: kick a background refetch so the real persisted rows
    //    replace the placeholders when they land.
    void refetch();
  } catch (e) {
    // FAILURE PATH: the POST itself failed — surface it, render nothing fake.
    toastError("Couldn't send message", { description: toApiError(e).message });
  }
}, [postChat, sessionId, refetch, beginStream]);
Why the order matters

The await postChat comes first on purpose: we need the server's real messageId before we can key the placeholder and target the stream. Everything after it is fire-and-forget for instant feel — setPending (paint), beginStream (tokens), void refetch() (reconcile in the background). The void says "I'm intentionally not awaiting this" — the UI doesn't block on the refetch; the overlay shows live tokens in the meantime.

Note the source of truth

The optimistic placeholder is not the source of truth — it's a temporary stand-in. The truth is the RTK Query chat-history cache. That's the whole discipline: keep one authoritative store (server cache), let the optimistic layer be explicitly disposable, and make the convergence automatic.

Argus

Reconciliation: overlay + placeholder drop

This is the cleverest part. The displayed messages array is derived (with useMemo) from three inputs — the server history, the live token map, and the optimistic placeholders — and the merge rules guarantee convergence.

Input 1 → 2: overlay live tokens onto the real row

// take server history; if a row is still streaming, swap in the live text
const overlaid = items.map((m) => {
  const live = liveText[m.id];
  if (live != null && (m.status === "streaming" || !m.content)) {
    return { ...m, content: live, status: "streaming" };  // live tokens win
  }
  return m;                                                // finished rows pass through unchanged
});

Input 3: synthesize placeholders — but only until the real row lands

if (!pending) return overlaid;

// THE KEY CHECK: has the real assistant row arrived from the server yet?
const assistantLanded = overlaid.some((m) => m.id === pending.assistantId);
if (assistantLanded) return overlaid;   // ✓ truth is here → DROP both placeholders

// otherwise show the optimistic pair (real id reused so the overlay targets it)
return [...overlaid, optimisticUser, optimisticAssistant];
The convergence guarantee

Because the optimistic assistant reuses the real messageId, the instant the server's row appears in items, assistantLanded flips true and the placeholders vanish — but the overlay keeps streaming live tokens onto that now-real row. There's no flicker: the text is continuous because both the placeholder and the real row render liveText[messageId]. The placeholder isn't "replaced" so much as it's "no longer needed."

  t0  send()         [user(opt)] [assistant(opt): ""]      ← placeholders only
  t1  first token    [user(opt)] [assistant(opt): "Based"] ← liveText grows
  t2  refetch lands  [user(real)][assistant(real): "Based on…"] ← placeholders dropped,
                                                                  overlay continues on real row
  t3  done frame     [user(real)][assistant(real): "…full answer"] ← refetch → persisted truth
Why derive instead of mutate

"I never imperatively edit a messages array — I keep three minimal sources (server cache, a liveText map, a pending object) and compute the rendered list from them in a useMemo. That means there's no way for the displayed state to drift out of sync with its inputs: change any input and the list recomputes correctly. It's the same 'derive, don't duplicate' principle as the run timeline's bySeq map."

Argus

Resume-on-refresh (the part most apps skip)

Optimistic UI handles the send. But what if you refresh the tab while the assistant is still generating? Naive apps lose the in-flight reply. Argus reattaches automatically.

// On mount, if history contains a still-"streaming" assistant row, reattach.
useEffect(() => {
  const streaming = items.find(
    (m) => m.role === "assistant" && m.status === "streaming",
  );
  if (streaming && !startedRef.current.has(streaming.id)) {
    beginStream(streaming.id, 0);   // reopen the SSE stream for that message
  }
}, [items, beginStream]);
Why this works — and why startedRef exists

The server persisted the assistant row with status: "streaming" and is still writing events for it. After a refresh, the history query returns that row; the effect spots it and reopens the stream from seq 0 (the backend replays via since_seq). The startedRef Set guards against reopening the same stream twice — without it, every re-render where items changes would spawn a duplicate EventSource. That ref is "have I already started this one?" memory that survives re-renders without causing them.

This is where SSE and optimistic UI fuse

Resume is impossible without both halves: optimistic UI defined the message id and rendering contract, and SSE's since_seq resumability lets a fresh page rejoin the exact stream mid-flight. "Refresh never loses or doubles the reply" is the sentence; seq-based replay + a persisted streaming row is the mechanism.

The server sends done with status + citations. The wrapper close()s (no reconnect). onDone clears activeId/pending, and a final refetch pulls the persisted content + citations — the live overlay is replaced by ground truth. If status === "failed", a toast fires instead.

The 20-second verbal version

"POST creates the rows and returns an id. I optimistically paint a placeholder pair so it's instant, open an SSE stream that appends tokens into a liveText map, and fire a background refetch. When the real rows land I drop the placeholders but keep overlaying live tokens onto the real row — no flicker. The done frame closes the stream and a final refetch swaps the live text for the persisted answer plus citations. A refresh mid-stream just finds the persisted streaming row and reattaches via since_seq."

Defense

Gotchas an interviewer will poke at

"What if…"How Argus handles it
…the network drops mid-stream?Wrapper reconnects with exponential backoff carrying since_seq; only un-seen events replay. The run hook flips isReconnecting to show a "lost" pill.
…the user refreshes mid-generation?The persisted streaming row is found on mount and the stream reattaches from seq 0 (server replays the tail).
…you get a duplicate event?Idempotent merge. The run hook dedupes by seq in a Map; chat deltas are append-only past the cursor. Same seq ⇒ ignored.

The cleanup question is the trap

The single most common follow-up is "where do you close the connection?" The answer is the useEffect cleanup function: useEffect(() => { const close = openStream(...); return () => close(); }, [deps]). Every subscription — SSE, WebSocket, interval, event listener — must be torn down on unmount or dep-change, or you leak the connection and run stale callbacks. Argus stores the teardown in a closeRef and also closes on unmount with a bare useEffect(() => () => closeRef.current?.(), []).

Why a ref for the teardown, not state

"The close function isn't rendered, and updating it shouldn't trigger a re-render — that's the textbook use of useRef: a mutable handle that survives renders without causing them. Same reason startedRef (dedupe set) and lastSeqRef (resume cursor) are refs — they're bookkeeping the UI doesn't paint."

Flashcards

Rapid-fire: SSE & optimistic UI

What is SSE in one sentence?

A browser standard for a one-way, server-to-client stream of UTF-8 text events over a single long-lived HTTP response, consumed with EventSource, with built-in reconnection and an event-id resume mechanism.

SSE vs WebSocket — when each?

SSE for one-directional server→client streams (feeds, progress, LLM tokens) — lighter, plain HTTP, auto-reconnect, survives proxies. WebSocket for full-duplex where the client also streams to the server (chat rooms, games, collab). Argus uses SSE because the client never pushes back over the channel — user actions are normal REST POSTs.

What does an SSE frame look like on the wire?

event: name, data: payload (we send JSON as a string), optional id: (the resume cursor) and retry:, terminated by a blank line. Content-Type: text/event-stream. A :-prefixed comment line is a heartbeat.

Why do named SSE events need addEventListener?

Because onmessage only fires for frames with no event: line (the default "message" type). Named events (event: delta) are dispatched to listeners registered for that name — so Argus registers one listener per event kind.

How does a dropped connection resume with no gap or duplicate?

Every event carries a monotonic seq. The client tracks the highest seen and reconnects with ?since_seq=N; the server replays only seq > N. The same seq makes the client-side merge idempotent (dedupe by seq). Reconnect uses capped exponential backoff and skips while offline.

What is optimistic UI and what's the risk?

Rendering the expected result before the server confirms, so the app feels instant. The risk is divergence on failure — so you need reconciliation (replace the guess with server truth) and a rollback/error path (toast + mark failed). Never leave a guess on screen as if it were real.

How do the optimistic placeholders get dropped without a flicker?

The optimistic assistant reuses the real messageId. When the server row appears in history, assistantLanded flips true and the placeholders are removed — but the live-token overlay keeps rendering onto the now-real row, so the text is continuous. The displayed list is a useMemo derived from (history, liveText, pending), so it can't drift.

Where do you close an SSE stream in React?

In the useEffect cleanup: useEffect(() => { const close = openStream(); return () => close(); }, [deps]). It runs on unmount and before every dep-change re-run. Forgetting it leaks the connection and triggers stale callbacks / setState-on-unmounted warnings. Argus also keeps the teardown in a closeRef.

Why is EventSource wrapped instead of used raw?

To control resume (cursor in the URL as since_seq rather than the native Last-Event-ID header), own the backoff, gate on navigator.onLine, make it SSR-safe (no-op without window), and make done authoritative so a finished stream never reconnects.
