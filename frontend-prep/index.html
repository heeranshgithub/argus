Argus Frontend

Next.js App Router & rendering

The App Router is the biggest "do you actually know modern React" topic. Know the server/client split cold.

Server Components vs Client Components

In the App Router, components are Server Components by default — they render on the server, ship zero JS to the browser, and can read data directly. A component becomes a Client Component only when it has "use client" at the top — needed whenever you use state, effects, event handlers, or browser APIs.

// frontend/src/app/providers.tsx — must be a client component
"use client";
import { useState } from "react";
// ... Redux Provider, ThemeProvider, etc. — all need the client
The rule of thumb to say

"Keep components on the server by default — they're free (no JS shipped). Push "use client" as far down the tree as possible, only where you need interactivity. In Argus the root layout is a server component; Providers (Redux + theme) is the client boundary, and pages that stream data are client components."

Argus Frontend · the crown jewel

RTK Query: server state done right

This is your strongest frontend talking point. RTK Query is a data-fetching + caching layer: you declare endpoints, it generates hooks, dedupes requests, caches by key, and invalidates by tags. You stop writing useEffect+useState fetch boilerplate entirely.

The setup — one API, endpoints injected per feature

// services/api.ts — the root API
export const api = createApi({
  reducerPath: "api",
  baseQuery: fetchBaseQuery({ baseUrl: env.NEXT_PUBLIC_API_BASE_URL, fetchFn: trackedFetch }),
  tagTypes: ["Session", "Run", "Report", "Chat"],  // cache tag vocabulary
  endpoints: () => ({}),
});
// store/index.ts — wire the API reducer + middleware into Redux
configureStore({
  reducer: { [api.reducerPath]: api.reducer },
  middleware: (get) => get().concat(api.middleware),  // caching/invalidation lives here
});

Queries, mutations, and the magic: tag-based invalidation

// services/sessions.ts
getSessions:   build.query({ query: () => "/api/sessions",
  providesTags: (r) => [{ type: "Session", id: "LIST" }, ...] }),  // this cache holds the LIST
createSession: build.mutation({ query: (body) => ({ url: "/api/sessions", method: "POST", body }),
  invalidatesTags: [{ type: "Session", id: "LIST" }] }),       // creating one → refetch the LIST
Explain it like this

"Each query provides tags describing the data it holds; each mutation invalidates tags. When I create a session, the mutation invalidates the Session/LIST tag, and RTK Query automatically refetches any mounted query that provided that tag. I never manually update the list — the cache graph does it. That's the whole difference between RTK Query and rolling your own fetching."

What you get for free

  • Auto-generated hooks: useGetSessionsQuery(), usePostChatMutation() — with isLoading, isError, data, refetch.
  • Request dedup & caching: two components asking for the same session share one request and one cache entry.
  • Normalized cache keyed by endpoint + args; automatic re-render only of subscribers.
  • Code-split endpoints: api.injectEndpoints() per feature file keeps concerns separated while sharing one cache.
"Why RTK Query over React Query / SWR / useEffect?"

"All solve server-state caching. I was already using Redux Toolkit, so RTK Query integrates with zero extra store — same devtools, same middleware. React Query is an excellent alternative and slightly lighter if you're not on Redux. The point I'd stress: server state and client state are different problems — server state needs caching, dedup, invalidation, and background refetch, which is exactly what these libraries give you and what naive useEffect fetching gets wrong (race conditions, waterfalls, stale data, no dedup)."

Streaming data into React (SSE)

The live features are where the interesting React patterns live. The challenge: a stream of events arriving over time must become consistent React state without losing or duplicating any.

The pattern: useReducer + a Map keyed by seq

// hooks/use-run-stream.ts — merge backfill + live events, deduped
interface State { bySeq: Map<number, WorkflowEvent>; runId: string | null; }

function reducer(state, action) {
  switch (action.type) {
    case "append":
      if (state.bySeq.has(action.event.seq)) return state;  // DEDUPE by seq → idempotent
      const bySeq = new Map(state.bySeq);
      bySeq.set(action.event.seq, action.event);
      return { ...state, bySeq };
  }
}
Why useReducer here, not useState

"The next state depends on the previous state and the update logic has branches (seed / append / clear). useReducer centralizes that transition logic, keeps it pure and testable, and avoids stale-closure bugs you get threading setState(prev => …) everywhere. Reach for useReducer when state transitions are complex or interdependent; useState for simple independent values."

The EventSource wrapper + manual reconnect

// lib/sse.ts — we drive reconnection to carry the resume cursor
let lastSeq = opts.sinceSeq ?? 0;
const handle = (ev) => {
  const event = JSON.parse(ev.data);
  lastSeq = Math.max(lastSeq, event.seq);   // track the cursor
  opts.onEvent(event);
};

The React lifecycle bit that matters

The hook opens the stream in a useEffect and returns a cleanup function that closes it: useEffect(() => { const close = openStream(...); return () => close(); }, [deps]). This is the single most-tested useEffect concept — every subscription must be torn down on unmount or dep-change, or you leak connections and get "setState on unmounted component" bugs.

Custom hooks & optimistic UI

Argus wraps each feature's behaviour in a custom hook — useRunStream, useChatStream — that returns a clean view model. This is the "logic in hooks, markup in components" separation interviewers like.

Optimistic UI: render before the server confirms

// hooks/use-chat-stream.ts — send()
const { messageId } = await postChat({ sessionId, content }).unwrap();
setPending({ userContent: content, assistantId: messageId, ... });  // optimistic placeholder pair
beginStream(messageId, 0);   // stream tokens immediately
void refetch();            // reconcile with the real server rows in the background
What optimistic UI is, and the risk

"Optimistic UI updates the screen assuming the operation will succeed, before the server responds, so the app feels instant. The risk is divergence if it fails — so you need a reconciliation/rollback path. In Argus the optimistic user+assistant placeholders are dropped once the real history rows land via refetch, and a failed reply surfaces a toast. The live tokens overlay onto the real row." This shows you know both the technique and its failure mode.

Hooks discipline you can name

Rules of Hooks: only call hooks at the top level (never in conditions/loops) and only from React functions — because React tracks hook state by call order. Custom hooks are just functions starting with use that compose other hooks. useCallback memoizes the send/retry functions so they're stable references passed to children.

Forms, Zod validation & env safety

Argus validates the create-session form with a Zod schema that mirrors the backend constraints — so bad input is rejected before it ever hits the API.

// schemas/session.ts — one schema, two outputs
export const sessionCreateSchema = z.object({
  companyName: z.string().trim().min(1, "Company name is required").max(200),

The Zod superpower to mention

"z.infer derives the TypeScript type from the runtime schema, so validation rules and types can never drift apart — one source of truth. I even validate NEXT_PUBLIC_* env vars with Zod at import time, so a misconfigured build fails fast with a clear message instead of erroring deep inside a request."

Controlled vs uncontrolled inputs (they'll ask)

Controlled: React state is the single source of truth — <input value={x} onChange={e => setX(e.target.value)}>. Uncontrolled: the DOM holds the value, you read it via a ref. Controlled is the default for validated forms (you can validate on every keystroke); uncontrolled is lighter for simple/native cases.

Tailwind, ShadCN & theming

Tailwind = utility-first CSS

Instead of writing CSS files, you compose utility classes in markup: className="min-h-full flex flex-col". Pros: no naming, no dead CSS, consistent spacing scale, styles co-located with markup. Con: verbose class strings. It's not inline styles — it's a constrained design-token system compiled to a tiny CSS file (unused classes are purged → small bundle).

ShadCN = copy-in accessible primitives

ShadCN isn't a dependency you import — it's components (Button, Dialog, Tabs…) you copy into your repo, built on Radix UI for accessibility (keyboard nav, focus traps, ARIA). You own and can edit them. Argus uses these for the consistent, accessible UI kit.

Dark mode with next-themes

// providers.tsx
<ThemeProvider attribute="class" defaultTheme="light" enableSystem={false} disableTransitionOnChange>

It toggles a class="dark" on <html>; Tailwind's dark: variants then apply. disableTransitionOnChange prevents a color-transition flash when switching.

Error handling & resilience

The frontend is hardened the same way the backend is — good to enumerate:

  • Route error/loading boundaries (error.tsx/loading.tsx) — a thrown error in a segment shows a recoverable fallback, not a white screen.
  • Offline banner that pauses SSE reconnect attempts when navigator.onLine === false (don't hammer a dead network).
  • De-duplicated toasts (capped stack) so a burst of errors doesn't spam.
  • Client-error reporter — uncaught client errors are sent to a backend sink.
  • Request-id correlation — a wrapped fetch records each response's x-request-id, so a UI error maps to the exact backend log line.
Error boundaries — the concept

"A React error boundary catches render-time errors in its subtree and shows a fallback instead of unmounting the whole app. In the App Router, error.tsx is a per-segment boundary (it must be a client component, and it receives a reset() to retry). Note: boundaries catch render errors, not async/event-handler errors — those you handle with try/catch + toasts, which is what the chat send() does."

React

The rendering model & reconciliation

If you understand why React re-renders, most React questions answer themselves.

Render → reconcile → commit

  1. Render: React calls your component function, producing a tree of elements (the "virtual DOM" — a lightweight JS description of the UI).
  2. Reconcile (diff): React compares the new tree to the previous one to find the minimal set of real DOM changes.
  3. Commit: React applies just those changes to the actual DOM.
Why keys matter

"When rendering a list, React uses the key to match elements across renders. A stable, unique key lets it tell which items moved, were added, or removed — without keys (or with index keys on a reorderable list) it mis-matches and you get wrong state/DOM reuse and subtle bugs. Never use the array index as a key for a list that can reorder or change length."

What triggers a re-render

  • State change (setState), or a parent re-rendering (children re-render by default).
  • Context value change (all consumers re-render).
  • A re-render does not mean a DOM update — reconciliation may find nothing changed.

The stale closure — the classic hooks bug

An effect/callback "captures" the variables from the render it was created in. If you omit a dependency, it keeps seeing the old value forever. Fix: list all referenced values in the deps array, or use a functional updater / ref. This is why Argus's hooks carefully list deps like [sessionId, refetch, beginStream] and wrap callbacks in useCallback.

The mental model — referential equality

"React compares props by reference. Objects/arrays/functions created during render are new every time, so a memoized child still re-renders unless you stabilize them with useMemo/useCallback. But these aren't free — premature memoization adds complexity and its own compute. Measure first (React DevTools Profiler), then memoize the proven hot path. Often the better fix is moving state down or splitting the component."

Note: React Compiler

If asked about the future: React 19's Compiler auto-memoizes, reducing manual useMemo/useCallback. Worth a sentence to show you're current — but the manual model is still what's tested.

React

State management landscape

Know the ladder and when to climb it:

  1. Local state (useState/useReducer) — default. Keep it here if only one component cares.
  2. Lift state up — when siblings must share, move it to the common parent and pass down.
  3. Context — for low-frequency global values (theme, auth user). Not for high-frequency updates (it re-renders all consumers).
  4. A store (Redux/Zustand/Jotai) — genuinely global, frequently-updated, or complex client state.
  5. A server-cache (RTK Query/React Query) — anything that lives on the server. Don't put server data in Redux by hand.
Context is not a state manager

Common interview trap: "use Context instead of Redux." Context is a transport (avoids prop-drilling), not a state container, and any value change re-renders every consumer. For frequently-changing global state it can cause perf problems Redux/Zustand avoid with selective subscriptions.

JavaScript

JavaScript core you must know

Closures

A function "remembers" the variables from the scope where it was defined, even after that scope returns. Powers hooks (state persists between renders via closures), module privacy, and event handlers. Also the source of the React stale-closure bug.

this

Determined by how a function is called, not where it's defined: method call → the object; standalone → undefined/global (strict); call/apply/bind → explicit; arrow functions → inherit this from the enclosing scope (why arrows are used for callbacks).

Prototypes

Objects delegate to a prototype chain for properties they don't own — JS's inheritance mechanism. class is syntax sugar over prototypes.

Promises & async/await

A Promise is a value that resolves later (pending → fulfilled/rejected). async/await is sugar over promises for readable sequential async code; await pauses the function until the promise settles. Know Promise.all (parallel, fail-fast), allSettled (wait all), race.

Quick hits

  • == vs ===: === no type coercion — always prefer it.
  • var/let/const: var is function-scoped + hoisted; let/const block-scoped with a temporal dead zone.
  • Event delegation: attach one listener to a parent and use bubbling, instead of N child listeners.
  • Spread/destructuring, optional chaining ?., nullish ?? — daily idioms.
Browser

The event loop

A near-guaranteed question. JavaScript is single-threaded; the event loop is how it does async without blocking.

  Call Stack (runs sync code)
       │  when empty, the loop pulls the next task ↓
  ┌────┴─────────────────────────────────────────┐
  │  Microtask queue   (Promises, queueMicrotask) │ ← drained FULLY first
  │  Macrotask queue   (setTimeout, events, I/O)  │ ← then ONE macrotask
  └────────────────────────────────────────────────┘
The rule that answers the trick question

After each macrotask (and after the current sync code), the engine drains the entire microtask queue before the next macrotask. So a Promise.then always runs before a setTimeout(…, 0) queued at the same time. Classic output ordering:

console.log("1");
setTimeout(() => console.log("2"), 0);   // macrotask
Promise.resolve().then(() => console.log("3")); // microtask
console.log("4");
// → 1, 4, 3, 2
Browser

How the browser renders a page

The critical rendering path

HTML → DOM; CSS → CSSOM; together → Render Tree → Layout (compute geometry) → Paint (fill pixels) → Composite (layer the painted pieces).

Reflow vs repaint (perf)

  • Reflow (layout): geometry changed (size, position, adding nodes) — expensive, recomputes layout. Triggered by reading offsetHeight mid-write, changing width, etc.
  • Repaint: visual-only change (color) — cheaper, no layout.
  • Composite-only (transform/opacity) — cheapest, GPU; prefer animating these.
render-blocking & why it matters

CSS is render-blocking (the browser won't paint until CSSOM is ready); <script> is parser-blocking unless async/defer. This is why critical CSS goes in <head> and scripts are deferred — and why frameworks like Next code-split and stream HTML.

Debounce vs throttle

Debounce: wait until activity stops, then fire once (search-as-you-type). Throttle: fire at most once per interval (scroll/resize handlers). Both prevent flooding expensive work — a frequent live-coding ask.

A core Next.js question. Know the tradeoffs of where and when HTML is built.

The tradeoff in one line

"It's freshness vs speed/cost. Static is fastest and cacheable but can go stale; SSR is always fresh but costs a server render per request; ISR splits the difference. Rendering

Hydration (and its pitfalls)

Hydration is React attaching event listeners and state to server-rendered HTML, making the static markup interactive. The HTML appears fast (good first paint); hydration makes it work.

Hydration mismatch — a real bug you handled

If the server-rendered HTML differs from what the client renders on first pass, React warns and may discard the server HTML. Causes: Date.now()/Math.random() in render, or reading localStorage/theme before mount. Argus sets <html suppressHydrationWarning> precisely because next-themes sets the theme class on <html> before React hydrates — an intentional, known mismatch on that attribute.

Tooling

TypeScript for frontend

Argus is fully typed. Be ready for the basics:

  • interface vs type: both describe shapes; interfaces extend/merge, types do unions/intersections. Either is fine; be consistent.
  • Generics: reusable typed containers — e.g. RTK Query's build.query<Result, Arg> types the response and the argument.
  • Union & narrowing: status: "streaming" | "complete" | "failed" + checks the compiler follows.
  • Derive types from runtime: z.infer<typeof schema> — Argus's pattern so types and validation never drift.
  • unknown vs any: unknown forces you to narrow before use (safe); any disables checking (avoid).
Flashcards

Rapid-fire: the Argus frontend

How do you fetch and cache server data?

RTK Query. I declare endpoints (queries/mutations) that get auto-generated hooks with loading/error/data. Queries provide cache tags; mutations invalidate them, so e.g. creating a session invalidates Session/LIST and any mounted list query refetches automatically. It also dedupes identical requests and shares one normalized cache. No manual useEffect fetching.

How do live workflow events become React state without dupes?

A custom hook opens an SSE stream and feeds events into a useReducer whose state is a Map keyed by the event's seq. The append action ignores any seq already present, so backfill + live merges are idempotent — no duplicates. The effect returns a cleanup that closes the stream on unmount/dep-change.

Server Components vs Client Components — when each?

Server Components are the default: they render on the server, ship no JS, and can read data directly — use for static/shell content. Add "use client" only where you need state, effects, event handlers, or browser APIs. Push the client boundary as far down as possible. In Argus the layout is server; Providers and streaming pages are client.

What's optimistic UI and how did you handle failure?

Rendering the result before the server confirms, so the app feels instant. On send I show optimistic user+assistant placeholders and start streaming immediately, then refetch to reconcile with the real rows (placeholders drop once they land). A failed reply surfaces a toast and is marked failed — so the optimistic state always converges with the truth.

Why useReducer over useState in the stream hook?

Because the next state depends on the previous one and the transitions branch (seed/append/clear with seq-dedupe). useReducer centralizes that as a pure, testable function and avoids stale-closure bugs. useState is for simple independent values.

How does one Zod schema validate forms and types?

I write the schema once (e.g. sessionCreateSchema) for runtime validation, and use z.infer<typeof schema> to derive the TypeScript type from it — so rules and types can't drift. I also validate NEXT_PUBLIC env vars with Zod at import time so misconfig fails the build fast.

Flashcards

Rapid-fire: fundamentals

What triggers a re-render, and does it always touch the DOM?

A state change, a parent re-rendering, or a consumed context value changing. No — a re-render produces a new virtual tree; reconciliation diffs it against the previous one and only commits actual DOM changes. A re-render with identical output touches no DOM.

Output order: log, setTimeout 0, Promise.then, log?

Sync logs first, then all microtasks (the Promise), then macrotasks (setTimeout). So 1, 4, 3, 2. The loop drains the entire microtask queue before the next macrotask.

useMemo vs useCallback?

useMemo caches a computed value; useCallback caches a function identity (useCallback(fn,d) ≈ useMemo(() => fn, d)). Both stabilize references so memoized children don't re-render. Don't overuse — memoization has its own cost; measure first.

Flexbox or Grid?

Flexbox for one-dimensional layout (a row or column — navbars, centering, distributing space). Grid for two-dimensional layout (rows and columns together — page layouts, card galleries). They compose: grid for the page, flex inside cells.

Why is the array index a bad list key?

If the list reorders, inserts, or deletes, index keys make React mis-match elements across renders — it reuses the wrong DOM/state, causing visual bugs and lost input focus. Use a stable unique id from the data.

Debounce vs throttle?

Debounce waits until activity stops then fires once (search input). Throttle fires at most once per interval (scroll/resize). Both cap how often expensive work runs in response to rapid events.

CSR vs SSR vs SSG?

Where/when HTML is built. CSR: in the browser after JS (bad SEO, slow first paint). SSR: per request on the server (fresh, SEO-friendly, costs a render). SSG: once at build time (fastest, CDN-cacheable, can go stale; ISR rebuilds periodically). Tradeoff is freshness vs speed/cost.

How does React prevent XSS, and where's the hole?

React escapes strings interpolated into JSX, so user content can't inject markup. The hole is dangerouslySetInnerHTML — if you use it, sanitize the HTML (e.g. DOMPurify) and add a Content-Security-Policy.
